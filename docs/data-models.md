# Data Models — Hexalith.Parties

> Generated by BMAD *Document Project* (deep scan, 2026-06-02). Source of truth is the code under `src/Hexalith.Parties.Contracts/` (domain contracts) and `src/Hexalith.Parties.Server/` (aggregate behaviour). This document is a navigational map — when in doubt, read the cited files.

Hexalith.Parties is an **event-sourced, CQRS** domain service. There is **no relational schema and no ORM**. The "data model" is:

1. A single **event-sourced aggregate** (`Party`) whose authoritative state is a stream of domain events.
2. A set of **commands** that request changes and **events** that record them.
3. A set of **read-model projections** (`PartyDetail`, `PartyIndexEntry`) materialised by DAPR actors and stored in a DAPR state store (Redis locally).

The contracts layer (`Hexalith.Parties.Contracts`) carries **no infrastructure dependency** — it references only `Hexalith.EventStore.Contracts`. It must not require a database, vector store, graph provider, or temporal store (enforced by `Contracts.Tests` architectural fitness tests).

## 1. The Party aggregate

There is exactly **one aggregate**: `Party`. Person vs. Organization is **not** a class hierarchy — it is a `PartyType` discriminator plus an optional details value object.

- **Aggregate state:** `PartyState` — `src/Hexalith.Parties.Contracts/State/PartyState.cs` (`sealed class`, mutable, rehydrated by replaying events).
- **Aggregate behaviour:** `PartyAggregate : EventStoreAggregate<PartyState>` — `src/Hexalith.Parties.Server/Aggregates/PartyAggregate.cs`. All command handlers are `public static DomainResult Handle(TCommand, PartyState?)` methods discovered by reflection.
- **Stream key / aggregate id:** the `PartyId` string (not stored as a state field). Event-store aggregate identity is `{tenant}:parties:{partyId}`.

### `PartyState` fields

| Field | Type | Notes |
|-------|------|-------|
| `CreatedAt` | `DateTimeOffset` | Set to `UtcNow` on `Apply(PartyCreated)` (not taken from the event). |
| `Type` | `PartyType` | `Person` or `Organization`. |
| `IsActive` | `bool` | Defaults `true`; toggled by deactivate/reactivate. |
| `IsNaturalPerson` | `bool` | Derived on create from `OrganizationDetails.IsNaturalPerson`; changeable via `SetIsNaturalPerson`. |
| `DisplayName` | `string` `[PersonalData]` | Derived (see Display-name derivation). |
| `SortName` | `string` `[PersonalData]` | Derived. |
| `Person` | `PersonDetails?` | Set when `Type == Person`. |
| `Organization` | `OrganizationDetails?` | Set when `Type == Organization`. |
| `ContactChannels` | `IReadOnlyList<ContactChannel>` | Tracked by stable `Id`. |
| `Identifiers` | `IReadOnlyList<PartyIdentifier>` | Tracked by stable `Id`. |
| `ConsentRecords` | `IReadOnlyList<ConsentRecord>` | GDPR consent ledger. |
| `ErasureStatus` | `ErasureStatus` | `Active` → `ErasurePending` → `KeyDestroyed` → `Verified` → `Erased`. |
| `ErasedAt` | `DateTimeOffset?` | Set on `PartyErased`. |
| `IsRestricted` / `RestrictedAt` / `RestrictionReason` | `bool` / `DateTimeOffset?` / `string?` | GDPR Art.18 processing restriction. |

**Rehydration note (important when adding events):** `PartyState` declares no-op `Apply(...)` overloads for every *rejection* event (rejection events are persisted to the stream and replayed). Because the EventStore rehydrator matches events by **short-name suffix**, the rejection `Apply` methods are deliberately declared *before* the success ones (e.g. `PartyProcessingRestricted` must be discovered before `ProcessingRestricted`). `Apply(PartyMerged)` and `Apply(PartyEncryptionKeyRotated)` are intentional no-ops.

### Display-name derivation

`PartyCreated` and any details/org update are paired with a `PartyDisplayNameDerived` event computed by the aggregate (`DeriveDisplayName`):

- **Person** → display `"First Last"`, sort `"Last, First"` (falls back to display when a name part is empty).
- **Organization** → display and sort both `LegalName`.

This logic is mirrored in the projection handlers so replays converge idempotently.

## 3. Commands

All under `src/Hexalith.Parties.Contracts/Commands/` — `sealed record`s, each carrying `PartyId`. Commands do **not** declare which events they emit; that logic lives in `PartyAggregate.Handle(...)`. The "→ emits" column below is the observed mapping.

### Lifecycle / creation

| Command | Key fields | → Emits (success) / rejects |
|---------|-----------|------------------------------|
| `CreateParty` | `Type`, `PersonDetails?`, `OrganizationDetails?` | `PartyCreated` + `PartyDisplayNameDerived`; rejects `PartyCannotBeCreatedWithInvalidId/WithoutType/WithoutPersonDetails/WithoutOrganizationDetails` |
| `CreatePartyComposite` | + `ContactChannels[]`, `Identifiers[]` | atomic create + bulk add → `CompositeCommandResult` |
| `UpdatePartyComposite` | `PersonDetails?`, `OrganizationDetails?`, add/update/remove contacts & identifiers | bulk mutation → `CompositeCommandResult`; rejects `CompositeOperationConflict` |
| `DeactivateParty` | — | `PartyDeactivated` / reject `PartyCannotBeDeactivatedWhenInactive` |
| `ReactivateParty` | — | `PartyReactivated` / reject `PartyCannotBeReactivatedWhenActive` |

### Detail updates

| Command | Key fields | → Emits |
|---------|-----------|---------|
| `UpdatePersonDetails` | `PersonDetails` | `PersonDetailsUpdated` (+ derived name) |
| `UpdateOrganizationDetails` | `OrganizationDetails` | `OrganizationDetailsUpdated` (+ derived name) |
| `SetIsNaturalPerson` | `IsNaturalPerson: bool` | `IsNaturalPersonChanged` |

### Contact channels & identifiers

| Command | Key fields | → Emits / rejects |
|---------|-----------|--------------------|
| `AddContactChannel` | `ContactChannelId`, `Type`, `Value`[PD], `IsPreferred` | `ContactChannelAdded` (+`PreferredContactChannelChanged`) / reject `PartyCannotAddDuplicateChannel` |
| `UpdateContactChannel` | `ContactChannelId`, nullable `Type?`/`Value?`[PD]/`IsPreferred?` | `ContactChannelUpdated` / reject `ContactChannelNotFound` |
| `RemoveContactChannel` | `ContactChannelId` | `ContactChannelRemoved` / reject `ContactChannelNotFound` |
| `AddIdentifier` | `IdentifierId`, `Type`, `Value`[PD] | `IdentifierAdded` / reject `PartyCannotAddDuplicateIdentifier` |
| `RemoveIdentifier` | `IdentifierId` | `IdentifierRemoved` / reject `IdentifierNotFound` |

### GDPR commands (consent, restriction, erasure)

These add a `TenantId`. **See the discrepancy note in §6** — these are implemented and active despite the README's MVP notice.

| Command | Key fields | → Emits / rejects |
|---------|-----------|--------------------|
| `RecordConsent` | `ChannelId`, `Purpose`, `LawfulBasis`, `Source?` | `ConsentRecorded` / reject `InvalidConsentPurpose` |
| `RevokeConsent` | `ConsentId`, `Reason?`, `Source?` | `ConsentRevoked` / reject `ConsentNotFound` |
| `RestrictProcessing` | `Reason?`, `CorrelationId?` | `ProcessingRestricted` / reject `PartyProcessingRestricted` |
| `LiftRestriction` | `CorrelationId?` | `RestrictionLifted` / reject `PartyNotRestricted` |
| `EraseParty` | `TenantId` | `ErasePartyRequested` / reject `PartyErasureInProgress` |
| `MarkPartyEncryptionKeyDeleted` | `DeletedAt` | `PartyEncryptionKeyDeleted` |
| `MarkErasureVerified` | `VerifiedAt`, `VerificationReportId` | `ErasureVerified` |
| `CompletePartyErasure` | `ErasedAt` | `PartyErased` |
| `RotatePartyKey` | `NewKeyVersion`, `PreviousKeyVersion` | `PartyEncryptionKeyRotated` |

**Idempotency:** handlers are state-driven and return `DomainResult.NoOp()` where appropriate — e.g. `CreateParty` when state already exists; `AddContactChannel`/`AddIdentifier` when the id is already present (safe for MCP retries); `RecordConsent` derives a deterministic `ConsentId = "{channelId}:{purpose}"` and no-ops on an existing active consent; deactivate/reactivate/restrict no-op when already in the target state.

**Composite guard:** `PartyAggregate.MaxSubOperations` (default 100) bounds composite commands; they validate-all-then-emit (all-or-nothing) and report `applied` / `skipped` / `rejected` lists.

## 4. Events

All under `src/Hexalith.Parties.Contracts/Events/` — `sealed record`s. Two categories by base interface (both from `Hexalith.EventStore.Contracts.Events`): **state-change** events implement `IEventPayload`; **rejection** events implement `IRejectionEvent`. A `DomainResult` is all-success or all-rejection, never mixed.

### State-change events

`PartyCreated`, `PartyDisplayNameDerived`, `PersonDetailsUpdated`, `OrganizationDetailsUpdated`, `IsNaturalPersonChanged`, `ContactChannelAdded`, `ContactChannelUpdated`, `ContactChannelRemoved`, `PreferredContactChannelChanged`, `IdentifierAdded`, `IdentifierRemoved`, `PartyDeactivated`, `PartyReactivated`, `ConsentRecorded`, `ConsentRevoked`, `ProcessingRestricted`, `RestrictionLifted`, `ErasePartyRequested`, `PartyEncryptionKeyCreated`, `PartyEncryptionKeyDeleted`, `PartyEncryptionKeyRotated`, `ErasureVerified`, `PartyErased`, `PartyMerged` (v2 placeholder, no-op).

See **[event-subscribing.md](event-subscribing.md)** for the CloudEvents wire format and **[event-handler-patterns.md](event-handler-patterns.md)** for per-event consumer handler patterns (the canonical, hand-maintained reference). Notable points:

- Events are published by **Hexalith.EventStore** (persist-then-publish) as **CloudEvents 1.0**, topic `{tenant}.parties.events`, id `{correlationId}:{sequenceNumber}`.
- `type` / `eventTypeName` are usually fully-qualified .NET type names — consumers normalize to short names before dispatch.
- `PersonDetails` fields and contact `Value`/identifier `Value` are `[PersonalData]` (decrypted at publish time; encrypted at rest under crypto-shredding).
- `PartyEncryptionKeyCreated` has **no** `Apply` override (emitted, not applied to party state).

### Rejection events (`IRejectionEvent`)

Published when commands fail (architectural Decision D3). [event-handler-patterns.md](event-handler-patterns.md) documents the **13 MVP** rejection events. The code additionally contains GDPR-scoped rejections — the full set found in `Contracts/Events/`:

- **Marker-only:** `PartyCannotBeCreatedWithInvalidId`, `PartyCannotBeCreatedWithoutType`, `PartyCannotBeCreatedWithoutPersonDetails`, `PartyCannotBeCreatedWithoutOrganizationDetails`, `PartyCannotBeDeactivatedWhenInactive`, `PartyCannotBeReactivatedWhenActive`, `PartyCannotAddDuplicateChannel`, `PartyCannotAddDuplicateIdentifier`.
- **With `Message?`:** `PartyNotFound`, `PartyTypeMismatch`, `ContactChannelNotFound`, `IdentifierNotFound`, `CompositeOperationConflict`.
- **GDPR / richer:** `PartyErasureInProgress`, `PartyProcessingRestricted`, `PartyNotRestricted`, `ConsentNotFound`, `InvalidConsentPurpose`.
- **Validation:** `PartyCommandValidationRejected` — `CommandType` + `Failures: IReadOnlyList<PartyValidationFailure>` (each `PropertyName` + `ErrorCode`; raw values/messages deliberately excluded to avoid PII leakage).

## 5. Read models (CQRS query side)

Read models are materialised by DAPR actors from the event stream and stored per-actor in the DAPR state store. Both carry **freshness metadata** so the gateway can surface stale/degraded reads.

### `PartyDetail` — full per-party view

`src/Hexalith.Parties.Contracts/Models/PartyDetail.cs`. Built by `PartyDetailProjectionActor` (`src/Hexalith.Parties.Projections/Actors/`). Fields: `Id`, `Type`, `IsActive`, `DisplayName`[PD], `SortName`[PD], `PersonDetails?`, `OrganizationDetails?`, `ContactChannels[]`, `Identifiers[]`, `ConsentRecords[]`, `NameHistory[]`[PD], `CreatedAt`, `LastModifiedAt`, `IsRestricted`, `RestrictedAt?`, `IsErased`, `ErasedAt?`, `Freshness?`.

### `PartyIndexEntry` — lightweight searchable row

`src/Hexalith.Parties.Contracts/Models/PartyIndexEntry.cs`. Stored as a per-tenant `Dictionary<partyId, PartyIndexEntry>` in `PartyIndexProjectionActor`. Fields: `Id`, `Type`, `IsActive`, `DisplayName`[PD], `SortName`[PD], `CreatedAt`, `LastModifiedAt`, `IsErased`, plus `[JsonIgnore]` in-memory `SearchableContactChannels` / `SearchableIdentifiers`. **Erased parties are removed from the index** entirely (never appear in search).

### Supporting model types

| Type | Purpose |
|------|---------|
| `PartySearchResult` | `Party: PartyIndexEntry` + `Matches: MatchMetadata[]` + `RelevanceScore` |
| `MatchMetadata` | `MatchedField`, `MatchType`, `Score?` (MVP only emits `displayName`) |

For projection actor mechanics (checkpointing, idempotency, rebuild, batching), see **[architecture.md](architecture.md) §4** and `src/Hexalith.Parties.Projections/`.

## 6. GDPR & personal-data model

- **`PersonalDataAttribute`** (`Contracts/PersonalDataAttribute.cs`) marks GDPR-sensitive properties. Used by (a) crypto-shredding to encrypt those fields at rest and (b) log sanitization to keep PII out of logs.
- **Crypto-shredding subsystem** lives in `Hexalith.Parties.Security` (per-party AES-256-GCM key; key destruction = erasure). Field-level `$enc` JSON markers; format `json+pdenc-v1`; redaction to `json-redacted` after key destruction.
- **Two independent switches** — do not conflate them:
  - `Parties:Compliance:GdprFeaturesActive` (default **false**) — controls only the **MVP compliance warning** (startup log + `X-Hexalith-Parties-Mvp-Compliance-Warning` response header).
  - `Parties:CryptoShredding:IsEnabled` (default **true**) — controls the **encryption-at-rest / erasure feature** itself.

> ⚠️ **Discrepancy to be aware of.** The `README.md` "GDPR Notice" states crypto-shredding/consent/erasure are *not* in the MVP and arrive in v1.1. The **code already implements** consent, processing-restriction, and crypto-shredding erasure, wired into DI and **enabled by default** (`IsEnabled=true`). The MVP *warning* and the crypto *feature* are separate toggles. Validate intended production posture against `docs/deployment-security-checklist.md`; the default key store is `LocalDevKeyStorageBackend` (in-memory, dev-only) and a real KMS/secret store is required for production.

GDPR-related operational docs: [gdpr-erased-party-status.md](gdpr-erased-party-status.md), [gdpr-key-rotation-and-shredding.md](gdpr-key-rotation-and-shredding.md), [gdpr-portability-export.md](gdpr-portability-export.md), [gdpr-processing-activity-records.md](gdpr-processing-activity-records.md), [deployment-security-checklist.md](deployment-security-checklist.md).

## 7. Serialization

- **No System.Text.Json polymorphism** is declared in the contracts (`[JsonPolymorphic]`/`[JsonDerivedType]`/source-gen contexts are absent). Event/command identity is the CLR type implementing the `IEventPayload` / `IRejectionEvent` marker interfaces; the rehydrator resolves events by **short-name suffix** (never `Type.GetType` on wire input — prevents arbitrary assembly load).
- **Wire convention:** camelCase property names, ISO-8601 dates, string enums, nulls omitted (`System.Text.Json` defaults). Result serialization in `Results/PartyCommandResult.cs` uses camelCase + `WhenWritingNull` + `JsonStringEnumConverter`.
- `[JsonConverter(typeof(JsonStringEnumConverter))]` on `ProjectionFreshnessStatus`; `[JsonIgnore]` on `PartyIndexEntry.SearchableContactChannels` / `SearchableIdentifiers`.

## 8. Reserved / future contract space

Present in the contracts but **not active in the MVP** (kept for additive forward-compatibility — do not document as live behaviour):

- `PartyMerged` event + `Apply(PartyMerged)` — **v2** placeholder (FR37).
- `TemporalNameResult`, `NameHistoryEntry`, `PartyDetail.NameHistory` temporal name-as-of reads — reserved; **no** query endpoint, client method, or MCP tool resolves a name as-of a timestamp (see [deferred-search-and-temporal-queries.md](deferred-search-and-temporal-queries.md)).
- `PartySearchMode` semantic/graph/hybrid values — wider than the wire allowlist; reserved for the optional Memories-backed search (see [memories-backed-party-search.md](memories-backed-party-search.md)).
