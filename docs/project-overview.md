# Project Overview — Hexalith.Parties

> Generated by BMAD *Document Project* (deep scan, 2026-06-02).

## Purpose

**Hexalith.Parties** is a ready-to-deploy **party-management domain service** for people and organizations — the party/contact data backbone (contact channels + identifiers) behind your own application logic. It is **not** an authentication provider, CRM, or identity server.

Public command and query traffic goes through **Hexalith.EventStore**; the `parties` service runs the domain actor host behind that gateway, and .NET consumers normally use the typed client package (`Hexalith.Parties.Client`).

## Key features

- **EventStore gateway ingress** — `POST /api/v1/commands` and `/api/v1/queries` with `Domain="party"`.
- **Parties actor host** — owns domain execution, projections, and DAPR actor hosting behind EventStore.
- **Typed client package** — `IPartiesCommandClient` / `IPartiesQueryClient` hide the EventStore envelope.
- **Embeddable party picker** — Blazor / custom-element selector (`<hexalith-party-picker>`).
- **Admin portal** — FrontComposer-hosted Blazor admin with a full GDPR operations suite.
- **Separate MCP host** — `parties-mcp` exposes `create_party`, `get_party`, `find_parties`, `update_party`, `delete_party`.
- **DAPR event subscription** — downstream apps consume party events with idempotent handlers.
- **.NET Aspire** — one-command local topology (EventStore, Parties, Tenants, DAPR sidecars, Redis, optional Keycloak).

## Repository structure

Single cohesive solution. **Adopter-facing** packages (`Client`, `Contracts`, `Picker`, `AdminPortal`, `Mcp`, `ServiceDefaults`) are what consumers reference; **internal** projects (`Hexalith.Parties` host, `Server`, `Projections`, `Security`, `Testing`) are private to the actor host. Sibling submodules `Hexalith.EventStore` and `Hexalith.Tenants` are project references (initialise before building). Annotated tree: [source-tree-analysis.md](source-tree-analysis.md).

## Important notes

- **GDPR posture:** the README states GDPR features are not in the MVP (v1.1), but the **crypto-shredding subsystem is implemented and enabled by default** (`Parties:CryptoShredding:IsEnabled=true`), distinct from the MVP-warning switch (`Parties:Compliance:GdprFeaturesActive`). Default key store is in-memory/dev-only — see [data-models.md §6](data-models.md) and [deployment-security-checklist.md](deployment-security-checklist.md).
- **MCP exposes 5 tools** (`get_party_name_at` does **not** exist; temporal name queries are deferred — [api-contracts.md §3](api-contracts.md)).
- Do **not** store regulated EU personal data in the MVP; use synthetic data for evaluation.
