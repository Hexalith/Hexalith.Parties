# Project Overview — Hexalith.Parties

> Regenerated by BMAD *Document Project* (**full rescan**, deep, 2026-06-02) - updated after Epic 5 (`ConsumerPortal` consent/privacy) implementation on 2026-06-10.

## Purpose

**Hexalith.Parties** is a ready-to-deploy **party-management domain service** for people and organizations — the party/contact data backbone (contact channels + identifiers) behind your own application logic. It is **not** an authentication provider, CRM, or identity server.

Public command and query traffic goes through **Hexalith.EventStore**; the `parties` service runs the domain actor host behind that gateway, and .NET consumers normally use the typed client package (`Hexalith.Parties.Client`). The repo now also includes `parties-ui`, a Blazor Server browser UI/BFF that owns OIDC sign-in and hosts the Admin/Consumer UI shell.

## Key features

- **EventStore gateway ingress** — `POST /api/v1/commands` and `/api/v1/queries` with `Domain="party"`.
- **Parties actor host** — owns domain execution, projections, and DAPR actor hosting behind EventStore.
- **Typed client package** — `IPartiesCommandClient` / `IPartiesQueryClient` hide the EventStore envelope.
- **Parties UI/BFF** — `Hexalith.Parties.UI` provides the FrontComposer/FluentUI Blazor Server host, host-owned OIDC, role-gated Admin/Consumer shell, accessibility guardrails, and live freshness foundations.
- **Embeddable party picker** — Blazor / custom-element selector (`<hexalith-party-picker>`).
- **Admin portal** — FrontComposer-hosted Blazor admin records management with GDPR operations surfaces, including erasure certificate display and verification retry through the EventStore-fronted client.
- **Consumer portal** — FrontComposer-hosted `/me*` RCL for Consumer My Profile, Edit My Profile, consent, data export, erasure request/cancel, and processing-transparency views, using ConsumerPortal-owned ports adapted by the UI host to the fail-closed self-scoped accessor.
- **Separate MCP host** — `parties-mcp` exposes `create_party`, `get_party`, `find_parties`, `update_party`, `delete_party`.
- **DAPR event subscription** — downstream apps consume party events with idempotent handlers.
- **.NET Aspire** — one-command local topology (EventStore, Parties, Parties UI, Tenants, DAPR sidecars, Redis, optional Keycloak).

## Repository structure

Single cohesive solution. **Adopter-facing** packages (`Client`, `Contracts`, `Picker`, `AdminPortal`, `ConsumerPortal`, `Mcp`, `ServiceDefaults`) are what consumers reference; `Hexalith.Parties.UI` is the browser UI/BFF host for this product experience. **Internal** projects (`Hexalith.Parties` host, `Server`, `Projections`, `Security`, `Testing`) are private to the actor host. Sibling submodules `Hexalith.EventStore` and `Hexalith.Tenants` are project references (initialise before building). Annotated tree: [source-tree-analysis.md](source-tree-analysis.md).

## Important notes

- **GDPR posture:** the **crypto-shredding subsystem is implemented and enabled by default** (`Parties:CryptoShredding:IsEnabled=true`), distinct from the MVP-warning switch (`Parties:Compliance:GdprFeaturesActive`). The only key store registered by default is `LocalDevKeyStorageBackend` (in-memory, dev-only), so a production KMS or secret-store-backed key provider is required before real personal data is stored — see [architecture.md §8](architecture.md), [data-models.md §6](data-models.md), [deployment-security-checklist.md](deployment-security-checklist.md), and the README notice.
- **MCP exposes 5 tools** (`get_party_name_at` does **not** exist; temporal name queries are deferred — [api-contracts.md §3](api-contracts.md)).
- Do **not** store regulated EU personal data in the MVP; use synthetic data for evaluation.
