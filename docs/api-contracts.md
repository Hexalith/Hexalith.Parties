# API Contracts — Hexalith.Parties

> Regenerated by BMAD *Document Project* (**full rescan**, deep, 2026-06-02). Sources: `src/Hexalith.Parties.Client/`, `src/Hexalith.Parties.Mcp/`, `src/Hexalith.Parties/` (host ingress). Re-verified this rescan: `parties-mcp` exposes **exactly 5 tools** (no `get_party_name_at`). For the end-to-end walkthrough with copy-paste curl/PowerShell, see **[getting-started.md](getting-started.md)**.

## The boundary in one picture

```
 .NET consumer ──► IPartiesCommandClient / IPartiesQueryClient ─┐
 AI assistant  ──► parties-mcp (5 tools) ──► typed client ──────┤
 Any HTTP client ──────────────────────────────────────────────┤
                                                                ▼
                                   Hexalith.EventStore gateway  (PUBLIC)
                                   POST /api/v1/commands   Domain="party"
                                   POST /api/v1/queries    Domain="party"
                                   (authn, tenant validation, RBAC, routing)
                                                                │ DAPR service invoke "process"
                                                                ▼
                                   parties  actor host  →  POST /process  (INTERNAL only)
```

**Key rule:** the `parties` actor host has **no public command/query API**. All public traffic enters through the EventStore gateway. The only client-reachable endpoint on the `parties` service is the internal DAPR `POST /process` (locked down so only `eventstore` may call it). AI assistants connect to the **separate `parties-mcp` host**, not to `/mcp` on `parties`.

## 1. Public HTTP gateway (EventStore)

Owned by **Hexalith.EventStore**, not by this repo, but Parties defines the command/query *shapes*. Base URL = the `eventstore` gateway (the `Parties:BaseUrl` config value), **not** the parties actor-host URL.

### Commands — `POST /api/v1/commands`

Envelope (camelCase JSON):

| Field | Meaning |
|-------|---------|
| `messageId` | client-generated id (GUID "N") |
| `tenant` | tenant id (required) |
| `domain` | `"party"` |
| `aggregateId` | the `partyId` |
| `commandType` | fully-qualified .NET command type, e.g. `Hexalith.Parties.Contracts.Commands.CreateParty` |
| `payload` | the command object |
| `correlationId` | usually `= messageId` |

Response carries a `correlationId` (and optionally a `resultPayload` echoing the updated `PartyDetail`). **Acceptance is not read-your-write** — projections are eventually consistent.

**Command types accepted** (the `commandType` field): every command in [data-models.md §3](data-models.md) — `CreateParty`, `CreatePartyComposite`, `UpdatePartyComposite`, `UpdatePersonDetails`, `UpdateOrganizationDetails`, `SetIsNaturalPerson`, `AddContactChannel`, `UpdateContactChannel`, `RemoveContactChannel`, `AddIdentifier`, `RemoveIdentifier`, `DeactivateParty`, `ReactivateParty`, and the GDPR commands (`RecordConsent`, `RevokeConsent`, `RestrictProcessing`, `LiftRestriction`, `EraseParty`, …).

### Queries — `POST /api/v1/queries`

Envelope fields: `tenant`, `domain="party"`, `aggregateId`, `queryType`, `entityId`, optional `payload`. The gateway maps `queryType` → projection actor.

| `queryType` | `aggregateId` | Payload | Returns |
|-------------|---------------|---------|---------|
| `PartyDetail` | the partyId | — | `PartyDetail` |
| `PartyIndex` | `"parties"` | `{ page, pageSize, type?, active?, created/modified after/before }` | `PagedResult<PartyIndexEntry>` |
| `PartySearch` | `"parties"` | `{ query, page, pageSize, mode? }` | `PagedResult<PartySearchResult>` |

#### `PartySearch` wire-mode allowlist (fail-closed)

The `mode` payload field is a string. **Only** these values are honoured (everything else fails closed with `UnsupportedQueryType` *before* the projection read, echoing no tenant/query detail):

- omitted / `null` / empty → MVP display-name search
- `"Lexical"` → display-name lexical search
- `"DisplayName"` → alias of the same

Reserved future values (`"Hybrid"`, `"Semantic"`, `"Graph"`, `"Email"`, `"Identifier"`, `"TemporalName"`) are **rejected** in MVP. The in-process `PartySearchMode` enum is intentionally wider than this allowlist. See [deferred-search-and-temporal-queries.md](deferred-search-and-temporal-queries.md). When Memories-backed search is enabled, additional rich modes apply — see [memories-backed-party-search.md](memories-backed-party-search.md).

## 2. Typed .NET client — `Hexalith.Parties.Client`

`dotnet add package Hexalith.Parties.Client`, then `builder.Services.AddPartiesClient(builder.Configuration)`. Config section `Parties`:

```json
{ "Parties": { "BaseUrl": "https://<eventstore-gateway>", "Tenant": "tenant-a" } }
```

Registration (`Extensions/PartiesClientServiceCollectionExtensions.cs`) eagerly validates `BaseUrl` (absolute http/https) and `Tenant` (non-empty) — **throws at startup** if invalid — and registers three typed `HttpClient`s. The clients hide the EventStore envelope; **tenant travels in the envelope body**, not a header.

### `IPartiesCommandClient`

`src/Hexalith.Parties.Client/Abstractions/IPartiesCommandClient.cs` (impl `HttpPartiesCommandClient`). Every command has **two overloads**:

- `…Async(...) : Task<string>` — returns the **correlationId** ("fire").
- `…WithResultAsync(...) : Task<PartiesCommandResult<PartyDetail>>` — correlationId + optionally-enriched `PartyDetail`.

| Method (Async / WithResultAsync) | Parameters |
|----------------------------------|-----------|
| `CreatePartyAsync` | `CreateParty, ct` |
| `CreatePartyCompositeAsync` | `CreatePartyComposite, ct` |
| `UpdatePartyCompositeAsync` | `string partyId, UpdatePartyComposite, ct` |
| `UpdatePersonDetailsAsync` | `string partyId, UpdatePersonDetails, ct` |
| `UpdateOrganizationDetailsAsync` | `string partyId, UpdateOrganizationDetails, ct` |
| `SetIsNaturalPersonAsync` | `string partyId, SetIsNaturalPerson, ct` |
| `AddContactChannelAsync` | `string partyId, AddContactChannel, ct` |
| `UpdateContactChannelAsync` | `string partyId, UpdateContactChannel, ct` |
| `RemoveContactChannelAsync` | `string partyId, RemoveContactChannel, ct` |
| `AddIdentifierAsync` | `string partyId, AddIdentifier, ct` |
| `RemoveIdentifierAsync` | `string partyId, RemoveIdentifier, ct` |
| `DeactivatePartyAsync` | `string partyId, ct` |
| `ReactivatePartyAsync` | `string partyId, ct` |

`PartiesCommandResult<TPayload>` = `record (string CorrelationId, TPayload? Payload)`. For `partyId`-taking overloads the client re-stamps `command with { PartyId = partyId }` so the route id is authoritative. Result enrichment is best-effort and **fails closed** — you only get a `PartyDetail` when the gateway echoes a matching-id `resultPayload`.

### `IPartiesQueryClient`

`src/Hexalith.Parties.Client/Abstractions/IPartiesQueryClient.cs` (impl `HttpPartiesQueryClient`).

| Method | Returns | Notes |
|--------|---------|-------|
| `GetPartyAsync(string partyId, ct, requestCustomizer?)` | `PartyDetail` | party-detail projection |
| `ListPartiesAsync(page, pageSize, PartyType? type, bool? active, createdAfter?, createdBefore?, modifiedAfter?, modifiedBefore?, ct)` | `PagedResult<PartyIndexEntry>` | filtered list |
| `SearchPartiesAsync(string query, page, pageSize, ct, string? mode = null, string? caseId = null, requestCustomizer?)` | `PagedResult<PartySearchResult>` | display-name search |

`requestCustomizer: Func<HttpRequestMessage, CancellationToken, ValueTask>` lets direct consumers add auth/tenant headers before send.

> **No temporal "name-at" query exists.** Temporal name data is only the `NameHistory` list inside `PartyDetail`; there is no as-of resolution method (reserved — see [deferred-search-and-temporal-queries.md](deferred-search-and-temporal-queries.md)).

### `IAdminPortalGdprClient`

`src/Hexalith.Parties.Client/AdminPortal/IAdminPortalGdprClient.cs` — a GDPR-focused client over the same gateway, used by the AdminPortal. Methods: `RequestErasureAsync` (→ `EraseParty`), `GetErasureStatusAsync` (derived from `PartyDetail`), `RestrictProcessingAsync`/`LiftRestrictionAsync`, `AddConsentAsync` (→ `RecordConsent`) / `RevokeConsentAsync`, `GetConsentAsync`, `ExportPartyDataAsync` (→ `ExportPartyData` → JSON download), `GetProcessingRecordsAsync` (→ `GetProcessingRecords`). Results map to an `AdminPortalGdprOutcome` enum (Accepted/Completed/ValidationRejected/Forbidden/…). Two methods are intentionally non-functional pending an EventStore contract: `GetErasureCertificateAsync` (faults 501 `ContractUnavailable`) and `RetryErasureVerificationAsync` (stub).

### Transport details

- Commands serialise to a private `EventStoreCommandRequest` record (`messageId`, `tenant`, `domain="party"`, `aggregateId`, `commandType = typeof(T).FullName`, `payload`, `correlationId`) → `POST api/v1/commands`.
- Queries serialise to `Hexalith.EventStore.Contracts.Queries.SubmitQueryRequest` (tenant, domain, aggregateId, queryType, projectionType `party-detail`/`party-index`, payload, entityId, projectionActorType) → `POST api/v1/queries`.
- **JSON options:** camelCase, ignore-null, `JsonStringEnumConverter`.
- **Correlation:** client-generated `messageId = Guid("N")`, used as both `MessageId` and `CorrelationId`. No explicit idempotency header.
- **Errors:** RFC 9457 `problem+json` parsed into `PartiesClientException(Status, Title, Type, Detail, CorrelationId)`; `Detail` is **redacted** when it matches sensitive tokens (payload/token/authorization/bearer/secret/password/dapr/redis/connection string/api-key).

## 3. MCP host — `parties-mcp`

`src/Hexalith.Parties.Mcp/`. A **separate** ASP.NET host (no DAPR sidecar) exposing Model Context Protocol tools over Streamable HTTP (`app.MapMcp()`, stateless). It forwards `Authorization`, `X-Tenant-Id`, `X-User-Id` from the request to the EventStore gateway via the typed client (`McpContextForwardingHandler`); it does **not** call Parties actors directly.

Client config:

```json
{ "mcpServers": { "hexalith-parties": {
  "url": "https://<parties-mcp>/mcp",
  "headers": { "Authorization": "Bearer <token>", "X-Tenant-Id": "tenant-a", "X-User-Id": "<user-id>" }
}}}
```

### Tools (exactly 5)

`Tools/PartiesMcpTools.cs` ([`McpServerToolType`]). Tool names in `Tools/PartiesMcpToolNames.cs`.

| Tool | Annotations | Behaviour |
|------|-------------|-----------|
| `get_party` | ReadOnly | `GetPartyAsync(partyId)`; returns `not_found` if the party is erased. |
| `find_parties` | ReadOnly | Display-name search (when `query` non-empty) or filtered list; params `query?, page, pageSize, type?, active?, created/modified after/before`. Clamps page≥1, pageSize 1–100. |
| `create_party` | Destructive | Forgiving AI-friendly input (`givenName`/`familyName` + aliases, `legalName`, `email`, `phone`, `vatNumber`/identifier, …). Generates a safe `partyId` if absent, builds a `CreatePartyComposite`, enforces payload-size limits. |
| `update_party` | Destructive | Patch semantics; route `partyId` authoritative. Merges via a read, builds `UpdatePartyComposite`; `active` applied as a separate (Re)activate command. |
| `delete_party` | Destructive, Idempotent | **Soft deactivation** (MVP delete intent) — explicitly *not* GDPR erasure. Idempotent when already inactive. |

> ⚠️ **`get_party_name_at` does NOT exist.** It is not implemented (temporal name resolution is deferred); the actual tool set is the five above. (The README + getting-started previously listed it as a "canonical tool" — corrected 2026-06-02.) Treat any reference to `get_party_name_at` as out-of-MVP.

Tool results use a `PartiesMcpToolResult` envelope (`status`, `category`, `code`, `message`, `toolName`, `correlationId?`, `data?`). `PartiesClientException` status codes map to categories (400/422 `validation_failed`, 401 `unauthorized`, 403 `forbidden`, 404/410 `not_found`, 409 `conflict`, 408 `timeout`, ≥500 `downstream_failed`). Every tool requires tenant+user context, else `missing_context`.

The MCP host binds `Parties:Mcp` options (`EventStoreGatewayBaseUrl`, default `http://eventstore`). It registers the typed clients as per-request transients whose tenant comes from the request context.

## 5. Authentication, tenancy & RBAC

EventStore owns **public** authentication, tenant validation, and RBAC at the gateway. The `parties` host additionally enforces defense-in-depth JWT bearer auth (`Authentication:JwtBearer`) and keeps a **local tenant-access projection** (`ITenantAccessService`, projection-side only — never on the gateway request path). Role→permission mapping (from Hexalith.Tenants events):

| Tenants role | Read | Write | Admin |
|--------------|------|-------|-------|
| `TenantReader` | ✓ | | |
| `TenantContributor` | ✓ | ✓ | |
| `TenantOwner` | ✓ | ✓ | ✓ |

The tenant-access projection **fails closed** (missing tenant/user, unknown/disabled tenant, non-member, insufficient role → deny) and is eventually consistent. Full details: [tenant-access-projection.md](tenant-access-projection.md). HTTP status meanings — `401` missing/invalid token; `403` gateway tenant/RBAC rejection (before domain execution).

## 6. Eventual consistency & freshness

Queries read **projection state**, not the write path. A query immediately after a command may miss the new party or return freshness metadata (`Rebuilding`/`Stale`/`Degraded`). Retry with bounded backoff, prefer `PartyDetail` by id for first-round-trip checks, and use the EventStore Admin UI for stream evidence. Responses may carry `X-Service-Degraded` / `X-Stale-Data-Age` (degraded reads) and `X-Parties-Search-Status` / `X-Parties-Search-Degraded-Reason` (Memories search). See [getting-started.md](getting-started.md#freshness-and-eventual-consistency).
