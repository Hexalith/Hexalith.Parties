## 1. Executive summary

Hexalith.Parties is a **ready-to-deploy party-management domain service** for people and organizations (contact channels + identifiers). It is built as an **event-sourced, CQRS** microservice that runs as a **DAPR virtual-actor host behind the Hexalith.EventStore gateway**. Consumers never call the actor host directly — public command/query traffic enters EventStore (`POST /api/v1/commands` and `/api/v1/queries`, `Domain="party"`), which authenticates, validates tenancy/RBAC, and routes the work to the `parties` host over DAPR service invocation.

It is **not** an auth provider, CRM, or identity server — it is the party/contact data backbone behind your own application logic.

**Integration surfaces:** a typed .NET client (`Hexalith.Parties.Client`), the `parties-ui` Blazor Server browser UI/BFF, a separate MCP host for AI assistants (`parties-mcp`, 5 tools), DAPR pub/sub event subscription for downstream read models, and embeddable Blazor UI (`Picker`, `AdminPortal`).

## 2. Architecture pattern

```
            ┌──────────── PUBLIC ────────────┐         ┌──────── INTERNAL (DAPR mesh) ────────┐
 consumer ─►│  Hexalith.EventStore gateway   │ invoke  │  parties  (this repo, actor host)    │
            │  /api/v1/commands  /queries    ├────────►│  POST /process                       │
            │  authn · tenant · RBAC · route │ "process"│                                      │
            └────────────────────────────────┘         │  ┌────────────── write side ───────┐ │
                      ▲   │ persist + publish           │  │ PartyDomainServiceInvoker        │ │
                      │   ▼ (CloudEvents)               │  │  → PartyAggregate.Handle(cmd)    │ │
            ┌─────────┴───────────┐                     │  │  → emits IEventPayload events    │ │
 subscriber │ DAPR pub/sub topic  │                     │  └──────────────┬──────────────────┘ │
   apps  ◄──│ {tenant}.parties.   │                     │     events       │                    │
            │ events              │                     │  ┌──────────────▼ read side ───────┐ │
            └─────────────────────┘                     │  │ PartyProjectionUpdateOrchestrator│ │
                                                         │  │  → PartyDetailProjectionActor    │ │
 .NET app ─► IPartiesQueryClient ──► gateway ──► query   │  │  → PartyIndexProjectionActor     │ │
 AI tool  ─► parties-mcp (5 tools) ─► typed client       │  └──────────────────────────────────┘ │
                                                         └──────────────────────────────────────┘
```

**Why this shape:** EventStore is a generic command/query/event substrate; Parties plugs in as a *domain service* registered under `Domain="party"`. This keeps authentication, tenant validation, RBAC, routing, persistence, and event publishing in EventStore, while Parties owns only domain rules and read-model projections. The boundary is enforced at the DAPR sidecar (access-control: deny-by-default; only `eventstore → POST /process`).

## 5. Write side — command processing

Entry: `POST /process` (DAPR invocation from EventStore) → `Program.cs` maps it to `IDomainServiceInvoker.InvokeAsync`.

1. **`PartyDomainServiceInvoker`** (`src/Hexalith.Parties/Domain/`, registered *before* `AddEventStoreServer` so it overrides the default) rejects non-`party` domains, resolves the command type **restricted to the contracts assembly** (no `Type.GetType` on wire input; ambiguous short names fail closed), runs the **FluentValidation** validator (→ `PartyCommandValidationRejected` on failure), and **decrypts** the current-state snapshot + prior event payloads via the crypto-shredding `IEventPayloadProtectionService` (redaction fallback after key destruction).
2. **`PartyAggregate.ProcessAsync`** (`EventStoreAggregate<PartyState>`) rehydrates `PartyState` by replaying events through `Apply(...)`, then dispatches the command to the matching static `Handle(cmd, state)`.
3. **Handlers** (`src/Hexalith.Parties.Server/Aggregates/PartyAggregate.cs`) enforce business rules in a guard cascade (null → id validity → idempotency no-op → type/required checks → emit events), then return a `DomainResult` of `IEventPayload`s (or rejection events). Success results may carry an enriched `PartyDetail` (`PartyCommandResult` / `CompositeCommandResult`).
4. EventStore **persists** the events to the aggregate stream and **publishes** them as CloudEvents (persist-then-publish; drain/recovery guarantees at-least-once).

Business-rule highlights: state-driven idempotency (`DomainResult.NoOp`), the erasure state machine (`ErasurePending → KeyDestroyed → Verified → Erased`), Art.18 restriction guards (with an Art.18(3) carve-out for consent ops), deterministic `ConsentId`, and the composite two-phase (validate-all-then-emit) with a `MaxSubOperations` cap.

## 6. Read side — projections (CQRS)

Events flow back into projections via **`PartyProjectionUpdateOrchestrator`** (registered for both `IProjectionUpdateOrchestrator` and `IProjectionPollerDeliveryGateway`). On delivery it reads the full stream from the EventStore `AggregateActor` (replay-from-zero; idempotency handled downstream), orders by sequence, unprotects payloads, and applies each event to two projection actors.

**Read models** ([data-models.md §5](data-models.md)):

- **`PartyDetailProjectionActor`** — per party (`{tenant}:party-detail:{partyId}`), materialises `PartyDetail`.
- **`PartyIndexProjectionActor`** — per tenant (`{tenant}:party-index`), a `Dictionary<partyId, PartyIndexEntry>` for list/search. Erased parties are removed from the index.

Each actor keeps a **last-processed sequence checkpoint** so replay-from-zero is idempotent, plus apply-level dedup (set semantics) so a crash between state and checkpoint writes still converges. The index actor **batches** writes (`BatchSize` default 50; flush via a `flush-batch` reminder). Both register an `auto-rebuild` reminder on detected state corruption, calling `ProjectionRebuildService` (which reads the persisted stream directly over the DAPR actor-state HTTP API, with resumable checkpoints).

**Query path:** `PartyDetailProjectionQueryActor` / `PartyIndexProjectionQueryActor` implement EventStore's `IProjectionActor.QueryAsync` and serve `PartyDetail`/`GetParty`, `PartyIndex` (paged+filtered), `PartySearch` (via `IPartySearchProvider`), and GDPR reads (`ExportPartyData` Art.20, `GetProcessingRecords` Art.30). Every read carries `ProjectionFreshnessMetadata`; stale/degraded reads fall back to a static last-known cache. Search defaults to `LocalFuzzyPartySearchProvider`; when `Parties:MemoriesSearch:Enabled=true` it swaps to `MemoriesPartySearchService` (Memories REST + local fallback) — see [memories-backed-party-search.md](memories-backed-party-search.md).

## 7. Eventing & subscribers

Events are published by EventStore as **CloudEvents 1.0**, topic `{tenant}.parties.events`, dead-letter `deadletter.{tenant}.parties.events`, id `{correlationId}:{sequenceNumber}`. Delivery is **at-least-once** — subscribers must be idempotent and always return `200 OK`. The repo ships a worked subscriber (`samples/Hexalith.Parties.Sample/PartyEventHandler.cs`) and exhaustive guidance:

- **[event-subscribing.md](event-subscribing.md)** — wire format, idempotency, ordering per broker.
- **[event-handler-patterns.md](event-handler-patterns.md)** — per-event handler patterns, the **mandatory `PartyErased` handler**, dangling-reference cleanup, tolerant deserialization.
- **[event-publishing.md](event-publishing.md)** — production broker config (Kafka/RabbitMQ/Service Bus), retry/circuit-breaker policies, dead letters, access control.

Parties is the **publisher only**; it does not consume its own party events. It *does* subscribe to **Hexalith.Tenants** events (`POST /tenants/events`, topic `system.tenants.events`) to maintain its local tenant-access projection.

## 8. Security & tenancy

EventStore owns public authn, tenant validation, and RBAC. Within `parties` (defense-in-depth):

- **JWT bearer** auth (`Authentication:JwtBearer`): OIDC `Authority` (prod) or symmetric `SigningKey` (dev, ≥32 chars); `MapInboundClaims=false`; RFC 9457 problem+json challenges. A single `"Admin"` authorization policy. `PartiesClaimsTransformation` normalises tenant claims to `eventstore:tenant`.
- **Tenant access projection** (`ITenantAccessService`, `src/Hexalith.Parties/Authorization/`): fed by Tenants pub/sub events; **fails closed**; role mapping `TenantReader`/`Contributor`/`Owner` → Read/Write/Admin. Eventually consistent (after a restart, in-memory projection starts empty and denies `UnknownTenant` until replay). **Used projection-side only — never on the gateway request path** (pinned by a fitness test). See [tenant-access-projection.md](tenant-access-projection.md).
- **Middleware pipeline** (`Program.cs`, order-sensitive): `CorrelationId → MvpComplianceWarning → ExceptionHandler → DegradedResponse → AuthN → AuthZ → CloudEvents`.

### GDPR / crypto-shredding

`Hexalith.Parties.Security` implements per-party **AES-256-GCM** field encryption (keyed off `[PersonalData]`), key lifecycle/rotation, erasure verification, and an append-only key-operation audit. Two independent switches:

- `Parties:Compliance:GdprFeaturesActive` (default **false**) → only the MVP warning (startup log + `X-Hexalith-Parties-Mvp-Compliance-Warning` header).
- `Parties:CryptoShredding:IsEnabled` (default **true**) → the encryption/erasure feature itself.

> ⚠️ **Discrepancy:** the README states GDPR features are "not in the MVP" (v1.1), but the crypto-shredding subsystem is implemented and **enabled by default** (`CryptoShredding:IsEnabled=true`). The README warning refers to the *compliance-warning* switch (`GdprFeaturesActive`), not the crypto feature.
>
> 🚨 **Production-data-loss risk (verify before any non-dev use):** `LocalDevKeyStorageBackend` (in-memory `ConcurrentDictionary`) is the **only** `IKeyStorageBackend` registered, and there is **no production-environment rejection guard and no startup "dev-only" warning**. Because crypto-shredding is ON by default, a production **restart silently destroys all key material** → all encrypted personal data becomes unrecoverable (effectively unintended mass erasure). The code anticipates an external KMS (it normalises a Vault-style `KeyNotFoundException`) but none ships. Provision a real KMS/secret store before storing any real personal data — see [deployment-security-checklist.md](deployment-security-checklist.md). (See also [data-models.md §6](data-models.md).)

## 9. Cross-cutting concerns

`Hexalith.Parties.ServiceDefaults` (Aspire service defaults) provides **OpenTelemetry** (logs/metrics/traces; OTLP exporter when `OTEL_EXPORTER_OTLP_ENDPOINT` is set; tracing source `"Hexalith.Parties"`), **structured JSON console logging**, **standard HTTP resilience**, **service discovery**, and the `/health` `/alive` `/ready` endpoints. The real DAPR health checks (`AddPartiesDaprHealthChecks` — sidecar, state store, pub/sub, projection actors, tenants, memories) are registered in `Program.cs`; readiness gates on sidecar + state store. `DegradedResponseMiddleware` surfaces `X-Service-Degraded` / `X-Stale-Data-Age` on degraded GETs.

**Logging discipline (privacy):** event payloads are never logged; projection logs carry only projection name + coarse counts, never party/tenant/actor ids (AC7). Personal data is `[PersonalData]`-tagged and kept out of logs.

## 13. Architectural decisions & gotchas

- **Single aggregate, discriminated** — `Party` with `PartyType` (Person/Organization), not a class hierarchy.
- **Gateway-fronted** — no public API on the actor host; only EventStore may call `POST /process`.
- **Rejection events are persisted & replayed** — `PartyState` declares no-op `Apply` overloads, ordered before success Applies (suffix-match rehydration).
- **Replay-from-zero on every delivery** — projections stay correct via per-actor sequence checkpoints + set-based apply.
- **Type resolution is allowlisted to the contracts assembly** — never `Type.GetType` on wire input.
- **Submodules referenced by path** (`Hexalith.EventStore`, `Hexalith.Tenants`) — now checked out as sibling directories in this working copy, but still resolved as **project references** (not NuGet), via a path-probing `HexalithEventStoreRoot` MSBuild property. A fresh clone must `git submodule update --init Hexalith.EventStore Hexalith.Tenants` (root-level only, **never** `--recursive`).
- **Apply-method ordering is load-bearing** — `PartyState` declares rejection-event `Apply` no-ops *before* success Applies (suffix-match rehydration); pinned by a fitness test. An auto-formatter that alphabetises methods would silently corrupt rehydration.

**Discrepancies & defects to reconcile (this rescan):**

- GDPR crypto-shredding enabled-by-default vs. README "not in MVP" (see §8 — includes the `LocalDevKeyStorageBackend` production-data-loss risk).
- **MCP exposes exactly 5 tools** (`create_party`, `get_party`, `find_parties`, `update_party`, `delete_party`); **`get_party_name_at` does not exist** (temporal name-as-of queries are reserved, not implemented). The README + `getting-started.md` previously listed a phantom 6th `get_party_name_at` tool — **corrected in this rescan.**
- **Projection-index rebuild key mismatch (bug):** the live `PartyIndexProjectionActor` reads/writes state key `{tenant}:party-index:default`, but `ProjectionRebuildService.GetIndexStateKey` uses `{tenant}:party-index:all` (`src/Hexalith.Parties.Projections/Services/ProjectionRebuildService.cs:460`). An index rebuild writes to a different key than the live actor reads — the two can diverge. (Detail-projection and manifest keys agree.)
- **Aspire SDK/package skew** (§3): AppHost SDK pinned `13.3.3` vs packages `13.4.0`.
- **AdminPortal GDPR client stubs:** `GetErasureCertificateAsync` and `RetryErasureVerificationAsync` are inert `ContractUnavailable` (501) placeholders pending an EventStore contract.
- **CI `contract-test` (Pact) job is unenforced** — the root Pact scripts don't exist, so it self-reports "CONCERNS" and is skipped; the quality gate passes it as `skipped`.
