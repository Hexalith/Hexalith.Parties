# Story 9.12: EventStore Admin UI Dapr Topology Correction

Status: done

## Story Boundary

This story corrects the post-Story 9.8 discovery that `eventstore-admin-ui` was incorrectly classified as a non-Dapr workload in the Kubernetes publish and validation contracts.

This story may change:

- `deploy/k8s/publish.ps1`
- `deploy/validate-deployment.ps1`
- `deploy/dapr/accesscontrol-eventstore-admin.yaml`
- `src/Hexalith.Parties.AppHost/DaprComponents/accesscontrol.eventstore-admin.yaml`
- generated Kubernetes manifests for `eventstore-admin-ui`
- deploy-validation tests
- Epic 9 topology documentation and planning artifacts

This story must not:

- remove the Admin UI Dapr sidecar
- grant wildcard Dapr access to the Admin Server
- add Dapr annotations to true non-Dapr workloads (`parties-mcp`, `redis`, `keycloak`, `falkordb`)
- weaken the Zot, namespace, or `-ConfirmContext` publish contracts

## Story

As an operator publishing Hexalith.Parties to Kubernetes,
I want `eventstore-admin-ui` modeled as a Dapr client-only workload,
so that Admin UI can invoke `eventstore-admin` through Dapr service invocation while true non-Dapr workloads remain protected from accidental sidecar injection.

## Discovery Context

During the clean publish retry, `publish.ps1` failed because `eventstore-admin-ui` carried Dapr annotations while the script still treated it as forbidden for Dapr. A source analysis showed the forbidden classification was stale:

- `Hexalith.EventStore.Aspire` attaches a Dapr sidecar to `eventstore-admin-ui`.
- `Hexalith.EventStore.Admin.UI` requires a Dapr sidecar at startup.
- Admin UI configures its Admin API client against the local Dapr HTTP endpoint.
- `DaprAppIdHandler` sends `dapr-app-id: eventstore-admin` for service invocation.

The correct topology is:

- full Dapr workloads: `eventstore`, `eventstore-admin`, `parties`, `tenants`, `memories`
- Dapr client-only workload: `eventstore-admin-ui`
- true non-Dapr workloads: `parties-mcp`, `redis`, `keycloak`, `falkordb`

## Acceptance Criteria

### AC1 - Publish preserves Admin UI client-only Dapr

Given `eventstore-admin-ui` is generated by Aspirate with Dapr annotations,
when `publish.ps1` patches generated manifests,
then it preserves `dapr.io/enabled: "true"` and `dapr.io/app-id: eventstore-admin-ui` on the pod template.

And it removes or rejects server-only Admin UI annotations: `dapr.io/app-port` and `dapr.io/config`.

And the publish guard still fails if true non-Dapr workloads carry any `dapr.io/*` annotation.

### AC2 - Validator understands full Dapr versus client-only Dapr

Given deployment validation runs against `deploy/k8s/`,
when a full Dapr workload is missing `enabled/app-id/app-port/config`,
then `K8sWorkload-MissingDaprAnnotations` is emitted.

And when `eventstore-admin-ui` is missing `enabled/app-id` or carries `app-port/config`,
then `K8sWorkload-MissingDaprAnnotations` is emitted.

And `parties-mcp`, `redis`, `keycloak`, and `falkordb` remain outside Dapr annotation requirements.

### AC3 - Admin Server ACL permits only the Admin UI caller

Given `eventstore-admin-ui` invokes `eventstore-admin` through Dapr service invocation,
when `accesscontrol-eventstore-admin` is applied,
then it allows caller app-id `eventstore-admin-ui` on the documented Admin API routes.

And it does not introduce wildcard caller app-ids or unrestricted `/**` operations in the production `deploy/dapr` configuration.

### AC4 - Planning and operator docs match the delivered topology

Given an operator reads the PRD, epics, Kubernetes architecture doc, getting-started guide, deployment guide, or deploy README,
when Dapr topology is described,
then `eventstore-admin-ui` is shown as Dapr client-only.

And the topology is described as 10 workloads with six daprd sidecars: five full Dapr workloads plus Admin UI client-only.

### AC5 - Live publish succeeds after namespace removal

Given the current `hexalith-parties` namespace is removed,
when `publish.ps1 -ConfirmContext kubernetes-admin@cluster.local` runs,
then the publish completes and all 10 workloads become Ready.

And registry-backed workloads use the same MinVer tag.

And `eventstore-admin-ui` runs with both containers ready: app container plus `daprd`.

## Dev Notes

- Admin UI is client-only because it initiates Dapr service invocation but is not a receiver that needs `dapr.io/app-port`.
- Receiver-side ACL belongs on `eventstore-admin`, not on the Admin UI sidecar.
- The production ACL allows `/api/v1/admin/**`, `/health`, and `/ready` for caller `eventstore-admin-ui`.
- The self-hosted AppHost Dapr access-control file remains allow-by-default because local caller identity is not consistently available without mTLS; it documents the intended caller without blocking local runs.
- Zot manifest deletion for `0.0.0-preview.0.500` was attempted before live publish but returned HTTP 403 for `parties-publisher`. That account can publish/update but not delete. The live publish still succeeded and verified all seven image manifests.
