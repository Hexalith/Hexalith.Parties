## What Went Well

- The team closed the binding design before implementation. Story 4.1 converted an open AR-Gap-Binding risk into an accepted admin-link ADR and concrete Story 4.2 criteria.
- Runtime Consumer access stayed fail-closed. `PartyIdClaimResolver`, `RoleLandingRedirect`, `NoPartyBinding`, and `ISelfScopedPartiesClient` remained the core seams, and runtime access still depends on exactly one IdP-emitted `party_id` claim.
- ConsumerPortal avoided a dependency inversion. The RCL owns `IConsumerProfileDataClient` and `IConsumerProfileEditClient`; the UI host registers scoped adapters to the self-scoped accessor.
- Story 4.2 review found and fixed important consistency issues: reconciliation authorization, rollback when IdP writes fail, and retained audit history after relinking suspended or removed records.
- Profile UI implementation stayed privacy-conscious: erased self profiles short-circuit PII fields, error copy is generic, profile values are not logged, and edit values match the stored/viewed values until the user types.
- Focused validation evidence was strong: ConsumerPortal/UI builds and compiled xUnit v3 test executables passed across the profile and binding surfaces.

## Challenges

- The original Epic 4 planning text used `ISelfScopedPartiesClient.GetPartyAsync(myPartyId)`, but the implemented safer contract is `GetMyPartyAsync()` with no caller-supplied party id. Story 4.4 corrected implementation guidance; durable docs needed the same cleanup.
- The shared status/freshness primitive question remains only partially resolved. Epic 4 avoided crossing dependencies by using ConsumerPortal-local display components, but no neutral shared UI package was introduced.
- The identity-binding store and IdP adapter are MVP in-memory host services. That fits the current implementation surface, but production audit durability and real IdP mutation need a later hardening path before real users depend on it.
- The local test harness remains unreliable for `dotnet test`/MTP and Playwright browser runtime in this sandbox. The team used focused builds, compiled xUnit runs, Playwright typecheck, and spec updates, but browser execution was blocked by socket permission errors.
- Story file hygiene improved but still required review fixes for File List completeness and source scans that must exclude generated/build output.

## Key Lessons

1. For Consumer self-scope, API shape matters as much as policy. No-caller-id methods (`GetMyPartyAsync`, `UpdateMyProfileAsync`) make the safe path harder to misuse than a method that accepts `partyId`.
2. RCL independence can coexist with host-owned security. ConsumerPortal ports plus UI-host adapters kept package boundaries clean while preserving the BFF self-scope choke point.
3. Provisioning state must stay consistent with the IdP claim. Store-only records are audit/reconciliation evidence, not runtime access grants; rollback and drift tests are necessary.
4. Review is most valuable when it checks failure ordering and artifact hygiene. The highest-impact Epic 4 fixes were around IdP failure rollback, reconciliation authorization, alert semantics, and file lists.
5. Environment-limited tests need precise labels. Compiled xUnit success is useful evidence; Playwright typecheck is not browser runtime proof.

## Next Epic Preview: Epic 5 Consumer - Consent, Data Export & Erasure

Epic 5 builds directly on Epic 4:

- `/me/consent` and `/me/privacy` already exist as protected ConsumerPortal routes.
- `ISelfScopedPartiesClient` already exposes no-caller-id GDPR self-service methods for consent, erasure, export, restriction, and processing records.
- The same ConsumerPortal port/adapter pattern should be used for Epic 5 instead of exposing host or gateway clients directly to the RCL.
- Existing Admin GDPR clients and backend contracts are available, but Consumer copy and UI state must remain distinct from Admin/DPO behavior.

Main dependencies and risks:

- Live OIDC `party_id` claim verification remains unproven outside static topology tests.
- Identity-binding audit persistence and real IdP mutation are still MVP/in-memory.
- Browser runtime proof remains blocked in this sandbox.
- Consumer privacy copy must preserve Epic 4's one-status-source and PII-free error/tombstone discipline.
