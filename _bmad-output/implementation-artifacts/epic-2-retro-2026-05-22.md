## What Went Well

### Breakthroughs worth replicating

1. **Handler/actor separation crystallized in Story 2.1** — pure handler returns `null` for no-mutation; actor preserves state. Eliminated a class of concurrent-state bugs. Held cleanly across 2.2–2.5 with zero rework — the strongest "pattern proven once, reused N times" result of the epic.
2. **Tenant isolation as positive proof, not audit (Story 2.6)** — fail-closed actor-key construction from authenticated EventStore context, proven-provenance gates for degraded reads, fail-closed cross-tenant detail probes. Replaces "audit after the fact" with "prove before returning." Became the template Epic 3 and Epic 4 consumed without modification.
3. **Additive typed freshness metadata (Story 2.7)** — `current | stale | degraded | rebuilding | unavailable` as a per-projection property, layered locally without forcing an EventStore contract rework. Allowed Epic 4 MCP to expose freshness without redesigning event sourcing.
4. **Wire-mode allowlist + explicit "reserved but not shipped" markers (Story 2.9)** — prevented the failure mode where reserved fields could accidentally become operational if a provider was added later. Pure-specification story; minimal code, high downstream protection.
5. **No-op idempotency as a cross-cutting contract (Stories 2.1, 2.2, 2.4)** — unchanged display name, duplicate stable-ID addition, missing-removal: all return success without mutation. Explicit in 2.1, surfaced as a gap in 2.2, enforced epic-wide thereafter.
6. **Two-pass review carried over from Epic 1 (Stories 2.1–2.5)** — first pass caught contract/scope gaps; second pass caught cross-cutting concerns (cancellation, PII, tenant probes). Measurable reduction in late-stage rework versus single-pass.

## What Didn't Go Well (systemic patterns, no blame)

### 1. Search-match-metadata taxonomy locked late

**Stories affected:** 2.5, 2.9

`MatchKind` and the reserved-mode set (email, identifier, semantic, temporal) were carried in the wire contract before their rejection/unsupported semantics were specified. Story 2.5 absorbed ~17 review patches as fuzzy matching, cross-tenant isolation in match output, erased-entry handling, and ranking semantics each surfaced a new clarification. Story 2.9 then had to retrofit a wire-mode allowlist and unsupported-status response. Root cause: search contract design completed under-specified — same pattern as Epic 1 rejection taxonomy, on a different surface. This is the recurrence of Action Item A1 from Epic 1 on a wider scope (see Lessons Applied below).

### 2. Cross-provider sort tie-breaker missing

**Story affected:** 2.4

`List<T>.Sort` (in-memory adapter) and `OrderBy.ThenBy` (LINQ providers) lacked a common final tie-breaker, producing non-deterministic order for equal display names across providers. Documented and deferred — not currently exposed in user-facing ordering — but **must land before any ranked search mode ships** (TD-1). Root cause: three query adapters implemented in parallel without forcing a common sort interface first.

### 3. Projection rebuild trust gates emerged in review, not design

**Story affected:** 2.8

Initial implementation had (a) a dead-letter probe that measured health-check actor liveness rather than real tenant projection state, and (b) a misleading cancellation message. The multi-step trust gate that ultimately shipped — replay → persistence → reload → health transition — was discovered during code review, not pre-specified. Root cause: rebuild semantics under-specified upfront; threat model for partial/concurrent rebuilds (what happens if a rebuild is killed mid-flight? if two rebuilds run for the same tenant?) was not part of the AC at story creation.

### 4. PII handling converged in retrofit rather than upfront policy

**Stories affected:** 2.1, 2.2, 2.3, 2.4, 2.5, 2.6

PII assertions, log-safety tests, erasure filtering, and `NameHistory` `[PersonalData]` marking each surfaced in separate review rounds 2.1–2.6. By 2.7–2.9 the patterns converged (diagnostic metadata only, erased entries invisible, `NameHistory` marked and erasable), but earlier stories were retrofitted. Root cause: no projection-layer PII policy document existed at the start of Epic 2 — Epic 1's `[PersonalData]` reflection tests covered the domain layer but did not transfer automatically to projections.

## Significant Discoveries (transferable design principles)

**Epic update required: NO** (Epic 3 already done; Epic 4 already done; Epic 7 contracts unaffected). The discoveries below are transferable principles, not amendments.

1. **Tenant isolation must be fail-closed at key-construction time, not audited after the fact.** Cross-tenant detail probes must return a bounded "not found" response — never an enumerable list, never a count-based existence signal. Story 2.6 proved the design discipline; any future read-side feature must design the actor-key shape and access control path before implementing handlers.
2. **Deferred capabilities require explicit wire-level guardrails, not just code-level TODOs.** Story 2.9's wire-mode allowlist + unsupported-status response is the model. Reserved fields and modes can accidentally become operational if a provider is added later without updating the gating layer — wire-level guardrails fail closed regardless.
3. **Projection freshness is orthogonal to EventStore consistency.** The store may be consistent while a projection is behind, rebuilding, or misconfigured. Treat projection health as an independent surface; don't conflate it with event-sourcing semantics. Informs Epic 4 SLA design and Epic 7 health monitoring UX.

## Key Takeaways (carry forward)

1. **The handler/actor separation pattern is the load-bearing read-side discipline.** Pure handler, null for no-mutation, actor preserves state, no-op idempotency on unchanged input. Repeat for every projection.
2. **Tenant isolation is a design-time discipline, not a review check.** Fail-closed actor-key construction from authenticated context, no caller-supplied tenant ids, no enumerable cross-tenant responses. Build the key shape and access path before the handler.
3. **Contract-surface taxonomy must be locked before implementation, not just for rejection events.** Epic 1 lesson A1 generalizes: any reserved set, enum, mode allowlist, or polymorphic shape pays a 14–17 patch tax if its unsupported/rejection semantics are clarified in review rather than design. This is Action B1.
4. **Projection freshness is its own layer.** Treat it orthogonally to EventStore consistency; expose typed transparency (`current | stale | degraded | rebuilding | unavailable`) additively without redesigning the store.
5. **Deferred capabilities need wire-level guardrails.** Reserved fields without unsupported-mode handling are a latent failure mode. The Story 2.9 wire-mode allowlist is the model.
6. **PII policy must be cross-layer, not domain-only.** Epic 1's domain `[PersonalData]` discipline did not auto-transfer to projections. Action B2 closes the gap before Epic 7 admin-console operators see any read-side data.
