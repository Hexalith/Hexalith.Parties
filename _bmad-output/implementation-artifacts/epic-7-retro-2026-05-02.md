## What Went Well

- Epic 7 turned the sample subscriber from a minimal demonstration into a realistic event-consumer reference.
- Publisher verification stayed aligned with EventStore ownership. The team tested and documented the contract rather than creating a parallel publishing implementation in Parties.
- DAPR configuration became explicit across local and deployment contexts: tenant-scoped topics, dead-letter topics, scopes, retry policies, access control, and broker-specific templates.
- The subscriber implementation covered practical consumer needs: deduplication by CloudEvents id, fallback id generation for local posts, tolerant deserialization, unknown event handling, selective handling, and order-tolerant projections.
- Review feedback improved real correctness. Fully qualified event type names, stale contact and identifier state, deployment-specific ordering guidance, and audit-log assertions were all corrected.
- Documentation moved beyond generic event concepts. `event-handler-patterns.md` covers each event type, rejection events, PartyMerged, PartyErased, dangling references, and tolerant reader patterns.
- PartyErased guidance was treated as a subscriber contract even before full erasure behavior ships in Epic 9. That gives consuming applications a clear compliance-facing obligation early.

Alice (Product Owner): "The product win is that event-driven integration is no longer vague. A subscriber developer can see what to subscribe to, how to handle it, and what not to assume."

Winston (Architect): "The architecture win is restraint. EventStore owns publishing mechanics; Parties documents and verifies the consumer contract."

## Challenges

- The CloudEvents contract had sharp edges. The real publisher emits fully qualified event type names, so sample dispatch and docs had to normalize names instead of assuming short event names.
- Broker ordering guidance could not be stated as a universal guarantee. Redis Streams, RabbitMQ, Kafka, and Azure Service Bus each require different deployment conditions for causal ordering.
- The subscriber sample shared static in-memory state across tests. xUnit parallel execution exposed that test isolation needed explicit serialization and cleanup support.
- Initial projection examples risked stale local state. Contact-channel removals, preferred contact changes, and identifier counts needed identifier-aware state tracking rather than simple increments or overwrite assumptions.
- Story 7.1 had to correct an overstated test claim. Publisher contract tests did not prove the full command pipeline or drain recovery, so completion notes were tightened to match actual coverage.
- PartyErased documentation had to be concrete without implementing Epic 9 behavior. The team handled that by documenting subscriber obligations and testing the documentation pattern itself.
- Deployment templates are useful but still operator-sensitive. Kafka partition routing and Azure Service Bus sessions need environment-specific configuration beyond static YAML examples.

Dana (QA Engineer): "The repeated issue was evidence precision. We had to keep separating what the test actually proves from what the architecture intends."

Charlie (Senior Dev): "That is healthy. It prevented false confidence around event delivery and ordering."

## Key Lessons

1. Event documentation must mirror the real wire contract.
   Evidence: Story 7.2 and 7.3 fixed fully qualified event type handling and aligned examples with publisher-generated CloudEvents.

2. Subscriber guidance needs executable patterns, not prose alone.
   Evidence: Story 7.2 added delivery, idempotency, tolerant deserialization, selective handling, and publisher-to-subscriber contract tests.

3. Broker-neutral pub/sub still requires broker-specific operator guidance.
   Evidence: Story 7.1 and 7.2 documented different ordering requirements for Redis Streams, RabbitMQ, Kafka, and Azure Service Bus.

4. Local read models should use idempotent state derivation where possible.
   Evidence: Story 7.2 replaced raw identifier increment/decrement behavior with identifier-ID tracking and recomputed contact projections after removals.

5. Future events need current handler patterns.
   Evidence: PartyMerged and PartyErased guidance lets consumers build tolerant handlers before those flows are fully active.

6. Documentation validation can be automated where the content is contractual.
   Evidence: Story 7.3 added tests that inspect `event-handler-patterns.md` for mandatory PartyErased guidance, normalized type guidance, broker-template links, and audit-log requirements.

## Significant Discovery Check

Amelia (Developer): "Epic 7 surfaced three discoveries that matter beyond this epic."

Significant discovery 1:

- The actual EventStore publisher contract emits fully qualified event type names.

Impact:

- Subscriber dispatch and documentation must normalize event names. Any future subscriber examples should include this from the start.

Significant discovery 2:

- Broker ordering is a configuration-dependent operational property, not a single product-level statement.

Impact:

- Epic 8 validation and operator docs should avoid generic "ordering is guaranteed" language unless the target broker configuration satisfies the documented conditions.

Significant discovery 3:

- PartyErased is a mandatory subscriber obligation even before full erasure implementation is active.

Impact:

- Epic 9 can build on a documented subscriber contract, but it must verify that the actual `PartyErased` event shape remains consistent with the handler guidance.
