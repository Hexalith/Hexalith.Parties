## What Went Well

- Story 7.1 created the target-destination ADR and release/rollback plan before
  code migration began. That prevented unowned cross-submodule changes.
- Commons adoption in Stories 7.2 and 7.3 kept Parties public surfaces stable.
  `PagedResult<T>`, correlation, ProblemDetails, health behavior, and local search
  semantics were preserved behind adapters or shims.
- Projection work in Stories 7.4 and 7.5 introduced an EventStore-facing adapter,
  kept replay-from-zero safety, wired freshness mapping into production reads,
  and retained rollback-only local paths where needed.
- Crypto work in Stories 7.6 and 7.7 used an ADR plus behavioral harnesses before
  migration. Review found and fixed a durable no-leak issue and an unbounded
  stack allocation in the new adapter.
- Senior review produced real quality gains: missing File List entries were fixed,
  false or weak evidence was corrected, and safety bugs were caught before closure.
- Story 7.8 gave an honest final readiness gate. It passed static deploy
  validation and focused security/projection/main assemblies, but did not claim a
  green release when broad validation was blocked.

## Challenges

- File List hygiene remained uneven. Reviews fixed missing files in several
  stories, which means the Epic 6 action item was not fully converted into a
  pre-review gate.
- Broad validation stayed sensitive to referenced-repo and environment blockers.
  The team improved at classifying blockers, but the release graph was still not
  green.
- Story 7.3 left a duplicated UI fixture normalization helper because routing it
  to Commons changed the UI build surface.
- Projection cleanup could not safely delete `LocalPartyProjectionPlatformAdapter`,
  `PartyProjectionPlatformAdapterMode.Local`, actor companion sequence keys, or
  `ProjectionRebuildService`.
- Crypto cleanup could not safely delete `PartyPayloadProtectionService` or the
  Parties key-management and erasure implementation because the EventStore-facing
  adapter still depends on them.
- Current root gitlink drift in Builds, EventStore, FrontComposer, Memories,
  PolymorphicSerializations, and Tenants remains a release-readiness blocker or
  pre-release hygiene task.

## Key Lessons

1. Adapter-first migration worked. The team could adopt shared primitives without
   losing rollback, public compatibility, or GDPR safety.
2. "Done" and "release-ready" are different states. Epic 7 is complete, but the
   final release gate remains blocked.
3. Focused tests need broad follow-up. Direct assemblies and focused builds are
   valuable, but they do not replace full solution, package, UI, deploy, and
   submodule-owner validation.
4. Security harnesses should run the full affected assembly, not only focused
   subsets. Story 7.6 review proved a focused subset can miss regressions.
5. Deletion needs stronger proof than adoption. Epic 7 proved adapters and
   compatibility; Epic 8 owns deletion-heavy domain-focus cleanup.

## Significant Discoveries

- Epic 7 should be classified as completed partial platform alignment, not full
  domain-focus refactoring. The stricter domain-focus target became Epic 8.
- Projection rebuild and actor companion state still protect replay, erasure, and
  rollback behavior. They must not be removed without replacement proof.
- Crypto-shredding remains Parties-policy-heavy. EventStore-facing metadata and
  typed unreadable outcomes improved, but production KMS and shared provider
  extraction remain outside Epic 7.
- The current root submodule graph can make broad solution validation misleading
  unless pointer drift is validated or reset deliberately.
- UI accessibility evidence is still a release gate: one direct UI test failed
  against the current FrontComposer-related navigation surface.
