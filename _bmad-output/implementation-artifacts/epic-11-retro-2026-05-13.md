## What Went Well

- Tenants became the explicit source of truth for lifecycle, membership, roles, and tenant configuration, while Parties kept ownership of party domain behavior and tenant-scoped data isolation.
- The team reused Tenants public client/testing seams instead of inventing Parties-local tenant lifecycle behavior.
- The access model became concrete and cumulative: reader for read/search, contributor for write, owner for administration.
- Denial behavior became consistent across REST and MCP with safe reason codes such as `missing-tenant`, `missing-user`, `unknown-tenant`, `tenant-disabled`, `not-member`, `insufficient-role`, and `tenant-state-stale`.
- Tests improved in the places that matter most for tenant safety: disabled tenant, removed user, missing membership, insufficient role, cross-tenant projection isolation, and authorization-before-side-effect ordering.
- Deployment validation moved Tenants integration from implicit configuration knowledge to actionable checks for pub/sub, topic, subscription, and secret-safe output.
- Documentation made the crucial distinction that a JWT tenant claim selects requested context, while active Tenants membership and role authorize access.

Alice (Product Owner): "The product win is simple: tenant authority is no longer ambiguous."

Winston (Architect): "The architecture win is that Parties consumes tenant authority without becoming the tenant-management service."

## Challenges

- Epic 11 crossed several boundaries at once: AppHost topology, Tenants client registration, DAPR subscription, local projection, REST/MCP authorization, deployment validation, and documentation.
- Story 11.1 initially left local bootstrap behavior ambiguous because Tenants can bootstrap a global administrator but does not automatically create tenant membership for Parties scenarios.
- Story 11.2 had to separate local projection freshness from request-path authorization. Missing projection state needed to fail closed without adding request-path Tenants polling.
- Story 11.3 tightened authorization late through review: strict `sub` then `oid` user extraction, payload tenant conflict rejection, call-order tests, and shared denial translation all emerged as necessary hardening.
- Story 11.4's validation scope grew beyond simple tests into production-style DAPR subscription and Tenants configuration checks.
- The later EventStore-fronted pivot changed the final placement of request-path tenant authorization: gateway-level tenant and RBAC validation moved to EventStore, while Parties retained projection-side tenant behavior.
- Deferred work accumulated around durable projection/deduplication, stale-state policy, deployment validation hardening, and broader topology migration notes.

Dana (QA Engineer): "The risk pattern was fail-open pressure. Claims, payload tenant fields, missing projection state, and deployment gaps all wanted to look harmless. The tests forced them to be explicit."

Paige (Technical Writer): "The documentation risk was authority drift. The guide had to say clearly: manage tenants in Tenants, not in Parties."

## Key Lessons

1. Tenant claims are context, not authorization.
   Evidence: Epic 11 repeatedly hardened flows so JWT tenant presence never grants access without active Tenants membership and the required role.

2. Authorization seams need shared vocabulary before they spread.
   Evidence: REST, MCP, tests, and troubleshooting all improved once denial reasons were stable and safe.

3. Eventual consistency must be documented as behavior, not hidden as an implementation detail.
   Evidence: The local Tenants projection can lag or be empty after restart, so Epic 11 made missing or unknown state fail closed and documented the operational repair path.

4. Deployment validation is part of security design.
   Evidence: Tenant access can look correct in code while DAPR subscription scopes, topic names, or integration config are wrong. Story 11.4 made those failures visible.

5. Cross-story boundaries need revisiting after architecture changes.
   Evidence: Epic 12 later moved gateway-level tenant enforcement to EventStore, changing how Story 11.3 should be interpreted for the final topology while preserving Story 11.2 projection-side value.

6. Reviews were productive because they resolved decisions, not only patches.
   Evidence: Bootstrap behavior, strict user-id extraction, singleton store expectations, endpoint authorization posture, and gateway ownership were all clarified through review records.

## Significant Discovery Check

Amelia (Developer): "Epic 11 did not invalidate the product direction. It exposed the exact split of responsibility that later made the EventStore-fronted pivot possible."

Significant discovery 1:

- Tenant authority cannot live in Parties if Parties is meant to validate the platform pattern.

Impact:

- Parties must consume Hexalith.Tenants state and preserve isolation, while Tenants remains the lifecycle and membership authority.

Significant discovery 2:

- Request-path authorization and projection-side filtering are different responsibilities.

Impact:

- After Epic 12, EventStore owns gateway tenant/RBAC validation, while Parties keeps projection-side tenant behavior and fail-closed read-model assumptions.

Significant discovery 3:

- Deployment configuration can break tenant safety even when code is correct.

Impact:

- Tenants pub/sub/topic/subscription validation belongs in release checks, not only in documentation.

Epic update required: No separate Epic 11 replan is needed. The important update is artifact reconciliation with the EventStore-fronted topology and follow-up triage for Tenants projection durability and validation hardening.
