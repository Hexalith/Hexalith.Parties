## Story Review

### 6.1 Activate Per-Party Personal Data Encryption

Validated the encryption foundation rather than inventing a parallel path. The story pinned derived display/sort names as personal data, redacted missing-key fallback behavior, encrypted snapshots and channels, mixed plaintext/encrypted replay compatibility, and publish-time decryption.

Evidence:

- `Hexalith.Parties.Security.Tests` passed 131/131.
- `EncryptionPipelineIntegrationTests` passed 6/6.
- The warning override remained necessary during this story because external warning-as-error failures were still present at that point.

### 6.2 Trigger Right-To-Erasure By Crypto-Shredding

Added validation before protected-state replay and proved key deletion makes encrypted personal data permanently unreadable through the payload protection service. The aggregate erasure lifecycle had explicit tests for accepted requests, idempotency, missing-party rejection, key-destroyed state, verification, and terminal `PartyErased`.

Evidence:

- Domain invoker validation passed 9/9.
- Key-management integration passed 8/8.
- Aggregate erasure tests passed 18/18.

### 6.3 Verify Erasure Across Internal Stores

Made erasure verification operationally useful by distinguishing `Pending`, `NotApplicable`, cleaned, failed, and skipped outcomes. Reports sanitize delegate and exception messages so operator-facing output does not leak personal data or raw backend details.

Evidence:

## Significant Discoveries

1. Crypto-shredding is not one feature; it is a cross-surface contract.

   Erasure affects aggregate behavior, protected payload replay, projections, exports, activity records, admin clients, subscriber notifications, and tenant rotation. The useful abstraction is not "delete a key"; it is "all surfaces return bounded, privacy-preserving truth after key destruction."

2. Operator-safe GDPR output needs its own design language.

   Store verification, portability status, processing records, erased status, and rotation status all converge on the same rule: bounded categories and counts are useful; raw backend text is unsafe.

3. Future event taxonomy should be independently guarded.

   Epic 5's contract-first bet was correct. Epic 6 proved it. The remaining weakness is fixture concentration, not taxonomy correctness.

4. Sprint workflows must follow sprint status, not filename prefixes.

   The stale Epic 6 and `6-*` artifacts make it easy to retro the wrong scope. This is no longer a theoretical renumbering risk.

## Key Takeaways

1. GDPR correctness has to be proven through every surface that can reveal, hide, transform, or summarize personal data.
2. Erased status is a product contract, not a crypto exception handling detail.
3. Bounded operator output is reusable architecture: categories, counts, and safe statuses beat raw backend messages.

5. Story artifacts need enough implementation/review detail to make retrospectives cheap and accurate. Story 6.10 is the better model.
