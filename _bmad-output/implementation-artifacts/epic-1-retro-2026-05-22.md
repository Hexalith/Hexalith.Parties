## What Went Well

### Breakthroughs worth replicating

1. **`PartyTestData` factory pattern** (Stories 1.2–1.9) — eliminated fixture duplication, made test intent legible, surfaced semantic boundaries (erasure vs. deactivation vs. restriction). Started ad hoc in 1.2, hardened across the epic.
2. **Two-pass review: party-mode + advanced-elicitation** (Stories 1.6–1.9) — party-mode caught coarse intent gaps (missing-party contract, FR69 scope); advanced-elicitation caught implementation traps (replay ordering, privacy leakage paths). Together caught ~85% of issues; either alone caught ~60%.
3. **Focused architectural fitness tests** (Story 1.1's `ClientArchitecturalFitnessTests`) — intent-based (no Dapr leakage) rather than version-pinned. Became the template for guardrail tests through the epic.
4. **Explicit deferred-decisions ledger** — every story named what it wasn't doing; tracked in `deferred-work.md`. Prevented silent scope creep and gave Epic 2 clarity on what's "real MVP" vs. "v1.1 reserved".
5. **Reflection-based `[PersonalData]` coverage tests** (Story 1.8) — caught regressions that would have leaked PII into telemetry. Template for Epic 6 GDPR work.
6. **`PartyDetail` canonical shape** (Story 1.9) — returning full state from mutations means clients don't issue redundant follow-up reads. Epic 2 projection-read API ended up mirroring this exact shape with zero impedance mismatch.

## What Didn't Go Well (systemic patterns, no blame)

### 1. Rejection-semantics inconsistency surfaced late

**Stories affected:** 1.3, 1.6, 1.7

Direct detail handlers initially returned `PartyTypeMismatch { Message = "Party does not exist." }` for null state. Story 1.7 had to unify it as `PartyNotFound` across create/update/lifecycle/contact/identifier. Root cause: rejection taxonomy was not defined upfront at Story 1.2. Cost: two-pass tax on five separate stories.

### 2. Privacy-safe assertion brittleness

**Stories affected:** 1.3, 1.4, 1.5, 1.7, 1.8

Composite checks like `result.Rejected.Any(x => x.Contains(personalValue))` failed spuriously when `Value` was empty string; null-forgiving operators hid nullability issues. Required dual-depth assertions (Rejected strings + persisted event Message) and explicit preconditions across the epic. Same assertion pattern rewritten five times.

### 3. Validator vs. aggregate id rule divergence

**Stories affected:** 1.4, 1.5

Composite validators required GUID-shaped contact/identifier ids; aggregate tests used readable ids. Inconsistency surfaced as review feedback in 1.4; formalized in 1.5. A Story 1.1 dependency-guardrails section listing boundary-vs-aggregate id divergence would have prevented this.

### 4. Derived-name personal-data markers initially missed

**Story:** 1.8

`PartyDisplayNameDerived.DisplayName`, `NameHistoryEntry.DisplayName`, `NameHistoryEntry.SortName` were unmarked with `[PersonalData]`. Caught only by reflection tests in Story 1.8. AC for `[PersonalData]` coverage should have explicitly listed derived-name fields in Story 1.2 or 1.3 when those fields first appeared.

### 5. Story 1.9 cross-module wire-DTO gap

**Story:** 1.9

`DomainServiceWireResult` initially dropped `DomainResult.ResultPayload` at the Dapr service-invocation boundary. EventStore's `CommandProcessingResult` already carried the field; gap was only at the wire DTO layer. 6+ days, 18 review findings (13 applied/addressed, 5 deferred). Cascaded to client, MCP, and gateway layers simultaneously. Lesson: validate wire DTO fields against actor result carriers **before** implementation begins.

## Recurring Review Feedback Themes

## Key Takeaways (carry forward)

1. **Establish rejection-event taxonomy and ordering invariants at Story 1.0 (or planning), not mid-epic.** Five stories paid two-pass tax for what should have been a one-time decision.
2. **Build shared test-data factories and assertion helpers on day one.** `PartyTestData` and the missing `AssertRejectionOnly` cost meaningful cycles by being late.
3. **Two-pass review (party-mode + advanced-elicitation) is the right gate for contract/wire/projection stories** — measurable ~85% issue catch vs. ~60% single-pass.
4. **Validate wire DTOs against actor result carriers before implementation when payloads cross Dapr service-invocation boundaries.** Story 1.9 cascaded 18 review findings from a single misalignment.
5. **Explicit deferred-decisions ledger works.** Zero silent scope creep across 9 stories. Keep this discipline.
6. **Downstream-epic carryover scan is the highest-value retro signal.** Epic 2 being already done turned this retrospective from speculation into evidence.
