## Significant Discoveries

1. Bounded blocker strings as cross-component contracts.

   The exact string `"Blocked on accepted EventStore-fronted Parties client/gateway contract"` introduced by 7.6 became the keystone reused by 7.7. The pattern of using a literal, copy-paste-stable string as a cross-component contract is worth promoting to architecture-level guidance (Action D3).

2. Capability surfaces require honesty tests.

   A capability that says `Available()` while its client reports `ContractUnavailable` is a latent UX-trust bug. Future capability surfaces need a self-consistency assertion - the capability advertised must match the client reality (Action D4).

3. Test-only regression pinning is a legitimate story shape.

   Stories 7.3, 7.4, and 7.10 each held the line by adding tests without touching production. This is healthy team maturity, not a lack of work. The pattern (existing implementation already meets ACs; story raises the floor with explicit regressions) is reusable for hardening epics.

4. Privacy guardrails should be source-scan-enforceable.

   Story 7.10's source-scan tests (no `MarkupString`, no browser storage APIs, no telemetry APIs in AdminPortal source) are reusable architecture, not Epic 7 trivia. The same pattern can guard Picker, MCP, and future UI surfaces.

## Key Takeaways

1. Bounded blocker strings are not a string - they are a contract. The exact text `"Blocked on accepted EventStore-fronted Parties client/gateway contract"` worked across 7.6 and 7.7 because it stayed bit-identical. Treat such strings as architecture, not as copy.

2. Capability surfaces lie unless tested. The production `GetGdprCapabilityAsync` advertised availability that its registered client did not provide. Adversarial create-story review found the latent bug. Future capability surfaces need self-consistency fitness tests.

3. Test-only stories are mature engineering. 7.3, 7.4, and 7.10 added safety without touching production. The pattern shows discipline, not stagnation.

4. Privacy compounds when reviews codify guardrails. 7.9's review corrections (no PII in ARIA, no party id in DOM ids) became 7.10's source-scan rules. Each review made the next story easier.

5. Cross-submodule drift can interrupt unrelated work mid-epic. Story 7.7's AppHost break was correctly out of scope, but stayed unresolved until Epic 9 v2. Cross-submodule edits remain a structural workflow hazard.

6. The Epic 6 C2 stale-artifact problem is now a confirmed pattern, not a one-off. Epic 7 stories 7.1, 7.2, 7.3 all had to manually disambiguate prior-plan artifacts. The disambiguation index (D1) is overdue.
