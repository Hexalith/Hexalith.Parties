## What Went Well

- GDPR became structural rather than decorative. `[PersonalData]` attributes, payload protection, key lifecycle services, and erased-party behavior are part of the system, not separate application code.
- The team reused Epic 8 operational foundations. Health/readiness, projection rebuild, graceful degradation, and DAPR state access patterns directly informed GDPR verification and Memories search behavior.
- Review remediation caught real risks before closure. Examples include tenant privilege escalation through query parameters, key-version cache bypass, audit concurrency, rotation rollback exception safety, empty tenant/key path validation, post-erasure replay failures, and shared-topology E2E pollution.
- The story split around search was healthy. Temporal name history stayed in the Parties projection where legal/audit semantics belong; rich discovery moved to Hexalith.Memories where semantic, hybrid, and graph retrieval belong.
- The erasure saga matured from endpoint behavior into a state machine. Aggregate transitions, persisted status records, certificates, verification reports, projection cleanup, and terminal erased state now align.
- Full-topology evidence improved late in the epic. Story 9.6 ultimately validated the full solution after isolating infrastructure-mutating Aspire tests and fixing the domain invoker registration order.
- The implementation preserved important boundaries. Contracts stayed free of Memories dependencies, domain code stayed encryption-unaware, and projection handlers remained testable pure logic where possible.

Alice (Product Owner): "The product win is that GDPR is now a real user journey, not a warning label. A DPO has APIs and evidence to work with."

Winston (Architect): "The architecture win is that privacy behavior is distributed across the right boundaries. No single controller is pretending to be the compliance system."

## Challenges

- DAPR secrets were not a full lifecycle API. Story 9.1 discovered that DAPR secret reads are portable, but create/rotate/delete require backend SDK abstraction.
- Key deletion changed the replay problem. Once keys are destroyed, internal erasure commands and projections must continue without accidentally demanding plaintext that no longer exists.
- EventStore behavior shifted under the epic. Persisted rejection events and suffix-based apply-method resolution required no-op rejection handlers to avoid state corruption during replay.
- GDPR status needs consistent vocabulary. Sprint status uses `done`, Story 9.2 says `Status: complete`, and Story 9.6 contains internal notes that still mention `review` despite the story and sprint status saying `done`.
- Memories-backed search introduced a deliberate plaintext-search-store tradeoff. The accepted design relies on reliable Memories cleanup during erasure; cleanup endpoint shape and mapping evidence must stay visible.
- Tier 3 tests exposed shared-topology risks. Several failures reproduced only after other E2E tests had polluted the shared topology, not in isolation. The eventual fixture isolation was the right fix.
- Temporal history has known hardening follow-ups. The story record defers event-envelope timestamp threading and name-history pagination/size policy.

Dana (QA Engineer): "The repeated pattern is that privacy correctness appears at intersections: key deletion plus replay, Memories plus erasure, restriction plus rejection events, topology health plus E2E sequencing."

Charlie (Senior Dev): "And the good news is that when the tests got adversarial, the system got better instead of just getting more patched."

## Key Lessons

1. GDPR behavior has to survive replay, not only request handling.
   Evidence: Story 9.6 needed post-erasure redaction fallback and domain/projection replay fixes so erased data stayed erased while system state could still advance.

2. External integrations need cleanup contracts as first-class acceptance criteria.
   Evidence: Memories search is valuable, but its GDPR acceptability depends on source-URI cleanup, blocked cleanup reporting, and hydration filters that drop stale or erased candidates.

3. Review findings are not overhead on privacy work.
   Evidence: Security issues, audit consistency, cache correctness, erasure saga progression, and topology isolation were materially improved by review passes.

4. Projection state is a compliance surface.
   Evidence: Name history, consent records, search indexes, erasure flags, sequence tracking, redacted replay, and cleanup reports all live on the read side.

5. Story scope should split when two architectural concerns compete.
   Evidence: The combined semantic-search/temporal-name story became cleaner when temporal name queries stayed in Parties and rich search moved to Memories.

6. Full-solution validation catches class-of-system issues that story-owned tests miss.
   Evidence: Story 9.6 story-owned tests passed before full solution failures exposed erasure/restriction/topology interactions.

## Significant Discovery Check

Amelia (Developer): "Epic 9 surfaced discoveries that affect Epic 10, but they do not invalidate Epic 10. They change the readiness bar."

Significant discovery 1:

- GDPR operations are state machines, not single buttons.

Impact:

- The admin UI must show pending, key-destroyed, verification-in-progress, verified, erased, failed, and blocked-cleanup states where applicable.

Significant discovery 2:

- Rich search can degrade or fall back.

Impact:

- The admin portal and embeddable picker should render search status and degraded reason instead of hiding the distinction.

Significant discovery 3:

- Memories-backed search contains personal data by design.

Impact:

- UI and operations docs must communicate cleanup-blocked states clearly; erasure completion cannot imply external search cleanup unless the verification report says so.

Significant discovery 4:

- Post-erasure replay and projection delivery are special paths.

Impact:

- Portal workflows should avoid assuming immediate projection freshness and should prefer persisted erasure status/certificate endpoints for compliance views.

Epic update required: No fundamental Epic 10 direction change is required. Epic 10 should be updated only to make Epic 9 state/status semantics visible and to avoid duplicating Tenants-owned administration.
