## What Went Well

- The MCP surface became real and complete. Epic 5 delivered five tools: `get_party`, `find_parties`, `create_party`, `update_party`, and `delete_party`.
- The translation-layer boundary held. MCP tools use commands, models, value objects, projection abstractions, and actor proxies without referencing domain event types or server aggregate internals.
- Review feedback materially improved the implementation. Story 5.1 added explicit MCP authorization, removed obsolete response-payload assumptions, and expanded search/index behavior.
- Tenant handling was solved in a way that matches the MCP transport reality. The team discovered that direct `IHttpContextAccessor` usage was unreliable in tool methods and moved tenant capture into `McpSessionContext` through session handling.
- `find_parties` became stronger than the minimum name-only path. Shared REST/MCP search logic now supports display-name, email, identifier, and multi-term matching with metadata.
- `create_party` handled AI-friendly input without weakening domain validation. Missing optional values, generated IDs, ISO date validation, and fallback response parity were all tightened through review.
- `update_party` preserved patch semantics by merging current projection state for whole-object details before command construction.
- `delete_party` became robust against stale projection responses and already-inactive parties.
- Story 5.4 converted architecture rules into executable safeguards: MCP event-type exclusion, allowed namespace validation, projection-handler DAPR exclusion, Contracts dependency checks, and Client boundary checks.

Alice (Product Owner): "The product win is that the AI-agent journey is no longer theoretical. The tools exist, they are named, and their behavior is documented through tests."

Winston (Architect): "The architecture win is that we did not let MCP become a second domain model. It translates; the aggregate decides."

## Challenges

- MCP session context was less straightforward than normal ASP.NET controller context. The team had to adapt to the SDK transport model and use session-level tenant capture.
- The `ModelContextProtocol.AspNetCore` API required experimental API handling and careful async hygiene.
- The response strategy from Epic 4 needed correction. Epic 5 had to remove obsolete `ResultPayload` assumptions before MCP and REST responses could align with the current EventStore contract.
- Search expectations expanded during implementation. Planning language emphasized display-name discovery, while MCP usefulness pushed the implementation toward email and identifier matching through shared index state.
- Eventual consistency affected response design. `create_party` needed fallback response assembly, and `delete_party` needed stale-projection handling to return the expected active-state signal.
- Patch semantics forced current-state reads before update command construction. That kept data safe but made the tool more complex than a simple parameter-to-command mapper.
- Tool tests and fitness tests required several test infrastructure patterns: tenant context reflection, actor proxy substitution, command payload capture, and assembly scanning.

Charlie (Senior Dev): "The hardest part was not writing a tool method. It was preventing tool convenience from turning into hidden domain logic."

Dana (QA Engineer): "And the tests had to cover both behavior and architecture. Unit tests alone would not catch a boundary violation."

## Key Lessons

1. AI-friendly does not mean domain-loose.
   Evidence: MCP tools accept forgiving input, but strict command validators and aggregate rules still own correctness.

2. MCP context handling must be designed around the transport model.
   Evidence: Story 5.1 moved away from direct `IHttpContextAccessor` assumptions and captured tenant context through session handling.

3. Shared search logic prevents REST and MCP drift.
   Evidence: Story 5.1 extracted `PartySearchResultsBuilder` and reused it across both surfaces.

4. Response parity matters for AI agents.
   Evidence: Story 5.2 aligned fallback display/sort name derivation with aggregate behavior; Story 5.3 forced inactive responses during delete projection lag.

5. Patch semantics require explicit merge rules.
   Evidence: Story 5.3 queries current state before constructing whole-object detail updates so unspecified fields remain unchanged.

6. Architectural rules need executable checks.
   Evidence: Story 5.4 added fitness tests that inspect references and method-body local variable types, not just public signatures.

## Technical Debt And Carry-Forward Items

- No dedicated no-PII logging regression guard is recorded for REST, MCP, command, or query paths.
- Composite-event projection replay coverage is still not clearly closed as a named test item.
- MCP tool naming remains split across older PRD wording and implemented architecture wording. Current implementation uses `find_parties`; docs and samples should consistently avoid stale `search_parties` wording unless explicitly framed as historical.
- Search scope exceeded the original MVP display-name-only wording. This is a useful capability, but planning and architecture notes should record the accepted email/identifier index behavior.
- MCP response behavior depends on projection freshness. Fallback and stale-response patterns exist, but should stay visible in future client and documentation work.
- D11 fitness tests protect type boundaries, but not semantic leakage. Reviews still need to check whether MCP code is making product/domain decisions instead of translating input.

## Significant Discovery Check
