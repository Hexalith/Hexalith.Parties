## What Went Well

- The shared-anchor convention landed across the intended surfaces. Claims,
  principal extraction, JSON options, projection names, actor ids, roles, policy
  names, text heuristics, export filenames, and display formatters now have
  canonical homes.
- `Contracts` remained infrastructure-free while still accepting BCL-only helpers.
  The one ASP.NET-bound exception, claims transformation, moved to the dedicated
  `Hexalith.Parties.Authentication` library as planned.
- Reviews caught meaningful risks before completion, including the Story 6.2
  deserialization regression and Story 6.4 GDPR `projectionType` routing bug.
- Several consolidations fixed real drift rather than just reducing duplication:
  payload protection regained string-enum behavior, GDPR export filenames now use
  one PII-free shape, projection rebuild keys now match live actor keys, and
  Consumer binding fails closed when tenant scope is missing or ambiguous.
- Focused test coverage improved around boundary fitness, package public API
  snapshots, authorization behavior, GDPR mapping, projection keys, and portal
  packaging.

## Challenges

- Story validation was repeatedly recorded as blocked when senior review later
  proved focused builds or direct xUnit runs could execute successfully with the
  repo's known fallback patterns.
- File List hygiene stayed noisy. Reviews fixed File List omissions in Stories
  6.1, 6.3, 6.4, 6.5, 6.6, and 6.7.
- Story 6.2 exposed the risk of treating a shared serializer as both canonical
  writer and strict reader without proving compatibility against the EventStore
  producer format and historical payloads.
- Line-ending churn appeared during review in more than one story and created
  avoidable diff noise.
- Full solution and package validation continued to be affected by workspace and
  dependency-graph blockers. The story records eventually distinguished focused
  story evidence from broader release readiness, but not consistently on first
  pass.

## Key Lessons

1. Consolidation stories need compatibility tests, not only shared-helper tests.
   Shared anchors can silently change producer/consumer behavior if historical
   wire shapes are not covered.
2. Senior review is paying for itself. It found real bugs, missing files, and
   overstated blockers before stories were closed.
3. Validation blockers need sharper classification. "Cannot run full solution
   build here" is different from "focused story evidence cannot run."
4. File List drift is a process smell. It appeared often enough to require an
   explicit pre-review check.
5. The Class A and Class B separation was useful. Epic 6 stayed in-repo while
   deferring platform extraction and deletion decisions to the architecture-gated
   Epic 7 path.

## Significant Discoveries

- The shared JSON story uncovered a critical read-compatibility issue: EventStore
  payloads and historical records require case-insensitive reading even while the
  canonical Parties writer emits camelCase, null-omitting, string-enum JSON.
- Projection-name adoption fixed a latent GDPR query bug: the legacy PascalCase
  `PartyDetail` projection type was rejected by the EventStore query gateway.
- Projection rebuild key consolidation fixed a mismatch between index rebuild
  state and the live index actor key.
- Claim extraction consolidation intentionally tightened Consumer binding:
  missing or ambiguous tenant scope now resolves unbound instead of binding with
  an empty tenant.

These discoveries do not require changing the Epic 7 direction. They reinforce
the adapter-first, parity-test, rollback-retaining approach already used by
Epic 7 and now feeding Epic 8 cleanup decisions.

## Next Epic Review

The next planned epic after Epic 6 was Epic 7: Platform Alignment - adopt
Commons/EventStore (Class B). This retrospective is being created after Epic 7
has already been marked done, so the preparation notes are historical and should
now be read as carry-forward guidance for Epic 8.

Epic 7 depended on Epic 6's discipline:

- Keep shared anchors stable while platform adapters are introduced.
- Preserve compatibility adapters and rollback paths until deletion proof exists.
- Treat projection, serialization, and crypto/security changes as compatibility
  migrations, not simple refactors.
- Separate focused story evidence from release-candidate readiness evidence.

Epic 7 final readiness later confirmed the same caution: local rollback paths and
some cleanup surfaces were preserved because deletion-safe parity was not fully
proved.

## Commitments

- Keep shared anchors in `Contracts` BCL-only unless an explicit exception is
  approved.
- Keep ASP.NET authentication normalization in `Hexalith.Parties.Authentication`,
  not `Contracts`.
- Do not treat a serializer or projection consolidation as complete without
  read/write compatibility proof.
- Do not mark validation as blocked until the focused fallback ladder has been
  attempted or the exact blocker is recorded.
- Do not let story File Lists depend on reviewer cleanup.
