## What Went Well

Amelia (Developer): "The team created a working vertical foundation rather than isolated pieces."

- The solution structure now mirrors EventStore conventions: `.slnx`, central package management, root build props, project layout, test projects, sample project, and strict dependency direction.
- Domain language was established early in Contracts: commands, events, value objects, `PartyState`, query models, rejection events, `[PersonalData]`, and forward-compatible event shapes.
- Aggregate logic stayed pure and synchronous, following EventStore conventions with `DomainResult`, `Handle`, and `Apply` patterns.
- Testing matured during the epic. The team moved from basic coverage to focused create/update/lifecycle test classes with reusable `PartyTestData`.
- API behavior became concrete: versioned routes, JWT tenant extraction, FluentValidation, ProblemDetails, domain rejection mapping, and anti-enumeration behavior.
- Local development became operable through AppHost, DAPR component configuration, ServiceDefaults, health/readiness endpoints, observability, and GDPR warning middleware.

Dana (QA Engineer): "The best quality signal is that review findings changed the implementation, not just the story text."

## Challenges

Charlie (Senior Dev): "Most issues were not caused by one person. They came from contract ambiguity, verification gaps, and documentation drift."

- Story 1.1 showed acceptance criteria drift early: `global.json` initially used the wrong SDK version, and `.gitignore` merge evidence was incomplete.
- Story 1.3 exposed missing domain guards for person/organization detail payloads, display-name derivation, and invalid `PartyId` handling.
- Story 1.4 repeated a related null-payload pattern in update handlers, where domain rejection should happen instead of allowing helper methods to throw.
- Story 1.5 initially had assertion-strength gaps: some tests checked event type/count but not enough payload detail.
- Story 1.6 had the largest review loop. Cross-tenant behavior, validator execution, missing-tenant response status, command authorization mapping, corrective-action metadata, handler naming, and response casing all needed reconciliation.
- Story 1.7 initially missed `subscription-parties.yaml`, readiness tagging, and stronger integration evidence for GDPR header and create/get behavior.
- Story records and file lists needed repeated correction to stay aligned with actual changes.

Alice (Product Owner): "The pattern is clear: we sometimes marked checklist items complete before the implementation evidence was strong enough."

## Key Lessons

1. Acceptance criteria must be executable, not just descriptive.
   Evidence: review found cases where task text said done but behavior or file evidence was incomplete.

2. Domain rejection paths need the same rigor as success paths.
   Evidence: null payloads, invalid IDs, type mismatches, and tenant failures repeatedly needed hardening.

3. Contract casing and error formats are API contracts.
   Evidence: `correlationId`, ProblemDetails shape, content type, and status code semantics all required explicit alignment.

4. Testing quality improved when assertions verified payload semantics.
   Evidence: Story 1.5 review strengthened tests beyond event count/type checks.

5. Retrospective-worthy design decisions should be written where future stories will see them.
   Evidence: opaque cross-tenant IDs returning `404` was a security decision, not just an implementation detail.

6. Story metadata is part of delivery quality.
   Evidence: file list drift and task text contradictions caused multiple review findings.

## Review Follow-Through Themes

Amelia (Developer): "Because there is no previous retrospective, this section captures recurring Epic 1 review themes that should become team habits."

- Guard clauses: validate command payload shape before helper methods or domain derivation logic run.
- Evidence-first completion: do not mark ACs or subtasks done until implementation, tests, and story file list agree.
- Security semantics: record intentional behavior when it differs from literal first-pass AC wording.
- Test depth: success tests should assert payload data, not only event existence.
- Operational readiness: health/readiness and DAPR component artifacts need direct verification, not assumption from configuration.

## Technical Debt And Carry-Forward Items

Dana (QA Engineer): "Nothing here blocks acknowledging Epic 1 as complete, but these items should shape the next work."

- Projection limitation: Story 1.6 uses direct aggregate actor state access for `GET /api/v1/parties/{id}` until projection work lands.
- Cross-tenant anti-enumeration: opaque foreign-tenant party IDs intentionally return `404`; richer projection-backed discrimination is deferred to later projection work.
- GDPR status: Epic 1 only adds the warning and structural `[PersonalData]` markers. Actual encryption, erasure, consent, and portability remain later-scope work.
- DAPR subscription and event-publishing posture: AppHost files exist, but subscriber experience and event-delivery validation remain future epic concerns.
- Story artifact discipline: file lists and task statements need a stricter closure pass before marking stories done.
