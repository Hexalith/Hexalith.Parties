## What Went Well

- Pure projection handlers worked as intended. Detail and index logic could be tested without DAPR, which made review fixes concrete and low-friction.
- The hybrid projection model held up. Full detail reads live in per-party actors, while list and search use the per-tenant index actor.
- Story 3.1 established a reusable handler and actor pattern, including tenant-scoped actor IDs and state keys.
- Story 3.2 added the index actor, partition strategy, configurable projection options, flush behavior, and actor interfaces needed for runtime registration.
- Story 3.3 made the read side usable through REST: list, search, projection-backed get-by-ID, pagination, filters, match metadata, and OpenAPI registration.
- Story 3.4 converted ambiguity into tests instead of leaving it as prose. Type filtering, date filtering, tenant isolation, empty index behavior, and page-boundary behavior all became executable checks.
- The review process found meaningful defects: missing DI registration, incomplete flush behavior, actor ID validation gaps, weak endpoint coverage, and overclaimed search scope.

Dana (QA Engineer): "The strongest pattern was that every serious review concern became either code, tests, or an explicit story clarification."

Winston (Architect): "The architecture was validated, especially D18. Pure handlers are doing real work for maintainability."

## Challenges

- The biggest issue was scope language drift. The original Epic 3 and PRD wording described search by name, email, and identifier, while architecture D2 explicitly deferred richer search to a later dedicated search engine.
- Story 3.2 initially implied searchable contact and identifier indexing, but the implemented `PartyIndexEntry` model intentionally does not store those fields.
- Story 3.3 initially lacked enough endpoint coverage for the new query behavior. Tests were added after review rather than being present at first closure.
- Story 3.4 needed multiple review rounds to settle the semantic gap around "email/identifier indexing." The final resolution documented v1.0 behavior, but the planning artifact still contains broader wording.
- Actor infrastructure had several runtime-readiness details that were easy to miss: actor interfaces, DI registration, option binding, flush behavior, and exact actor ID validation.
- Tenant isolation improved through tests, but the PRD target calls for stronger concurrent multi-tenant validation than the current story evidence demonstrates.
- No dedicated no-PII logging guard was added, despite being carried forward from Epic 2.

Charlie (Senior Dev): "The implementation did the right constrained thing for v1.0, but the story wording kept pulling us toward a larger search model."

Alice (Product Owner): "That is a product planning problem, not a developer problem. The roadmap language needs to distinguish v1.0 lookup from later search infrastructure."

Amelia (Developer): "Agreed. The system improvement is to keep planning artifacts synchronized with architecture decisions as soon as scope is clarified."

## Key Lessons

1. Projection scope must be stated in terms of the actual read model.
   Evidence: `PartyIndexEntry` supports display-name search and date filtering, not email or identifier search fields.

2. Pure handlers are the right unit of projection correctness.
   Evidence: Stories 3.1, 3.2, and 3.4 used handler tests to verify event-to-state behavior without actor infrastructure.

3. Actor wrapper behavior needs explicit runtime tests or review checklists.
   Evidence: Story 3.2 review found missing DI, options binding, actor interface registration, and flush implementation.

4. Search behavior needs negative tests as well as positive tests.
   Evidence: Story 3.4 added a test documenting that email queries return no matches in the current display-name-only model.

5. Tenant isolation must be proven through payload content, not just result counts.
   Evidence: Story 3.4 strengthened isolation tests with explicit returned-ID checks.

6. Story and PRD language must be revised when architecture intentionally narrows scope.
   Evidence: Epic 3 implementation aligns with D2, but older requirements still imply email and identifier search in MVP wording.

## Technical Debt And Carry-Forward Items

- Planning artifact mismatch: Epic 3 and PRD wording still imply email and identifier search in places, while implementation and architecture defer that richer search behavior.
- No executable no-PII logging guard for query paths was added.
- No concurrent multi-tenant isolation test suite is recorded, even though PRD/NFR language asks for stronger isolation evidence.
- Projection degradation behavior from D15 is not part of Epic 3 evidence; corruption handling and rebuild behavior remain future operational work.
- Projection rebuild tooling from D14 is not delivered in Epic 3; it is still an operational readiness concern.
- The index actor remains single-partition by implementation choice. The abstraction exists, but multi-key behavior is intentionally not proven.
