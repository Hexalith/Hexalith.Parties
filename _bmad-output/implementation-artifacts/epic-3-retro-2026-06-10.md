## Epic Summary

- Direct `/admin/parties/{id}/gdpr` route and safe detail-to-GDPR navigation.

## What Went Well

- The team kept the GDPR work in the existing AdminPortal route and `PartyGdprOperationsPanel`, avoiding duplicate page state and lower-level Razor client calls.
- Privacy constraints became stronger across the epic: typed-confirm values stayed in component memory, raw problem details and correlations were bounded, export filenames ignored server names, and processing records rendered stable metadata only.
- The D7 backend story found a simpler approved route than the original assumption: `GetErasureCertificate` uses the existing projection-query actor seam and retry uses the EventStore command path, so no EventStore submodule route or DAPR ACL expansion was needed.
- Review fixes were high-value and specific: stale dialog cleanup, pending-erasure mutation guards, bounded consent ledger rendering, stale export/correlation cleanup, capability fallback, 501 mapping, and certificate fallback behavior.
- Focused validation was consistent: AdminPortal, Client, Parties, DeployValidation, UI builds or direct xUnit assemblies passed where applicable; Playwright typecheck and discovery passed.

## Challenges

- Several planning docs still described D7 as pending stubs after implementation completed it, which would mislead future Epic 4/5 work unless corrected.
- Local `dotnet test` and Playwright runtime execution remained blocked by local IPC/socket permissions. The team worked around this with focused builds, direct xUnit v3 assemblies, TypeScript typecheck, and Playwright discovery, but browser runtime proof remains environment-bound.
- Full solution validation continued to hit unrelated sibling/submodule packaging warnings in `Hexalith.PolymorphicSerializations`, so story evidence had to stay focused and explicit.
- GDPR state is broad and easy to regress. Multiple review fixes involved stale operation state after party, tenant, capability, or erasure-status changes.
- The shared `StatusKind`/freshness primitive ownership question remains relevant for ConsumerPortal; AdminPortal stayed host-agnostic with local tested behavior.

## Key Lessons

1. Existing seams were enough for D7. The approved implementation used `PartyDetailProjectionQueryActor` plus the command domain service path instead of broadening EventStore or DAPR routing.
2. GDPR UI correctness depends on stale-state hygiene. Party, tenant, capability, erasure-status, and cancellation guards need regression tests around every operation that can outlive its initiating context.
3. Bounded display rules need to be tested at the component and fixture levels. Consent ids, channel ids, operator ids, server filenames, and certificate details can leak unless tests pin the intended surface.
4. Environment-blocked browser execution must be recorded precisely. Typecheck and spec discovery are useful evidence, but they are not browser runtime proof.
5. Planning assumptions need a cleanup pass after implementation. The D7 contract changed from "pending stubs" to implemented projection-query/command behavior.
