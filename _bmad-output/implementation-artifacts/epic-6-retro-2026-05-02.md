## What Went Well

- The client package delivered the promised .NET integration surface. `IPartiesCommandClient` and `IPartiesQueryClient` cover commands and queries while hiding DAPR, MediatR, server projects, and projection internals from consumers.
- Review feedback improved correctness instead of only polishing style. Story 6.1 fixed route/body `PartyId` normalization, eager `Parties:BaseUrl` validation, malformed success-payload handling, options binding, and restored NuGet graph inspection.
- The NFR28 dependency-count conflict was handled transparently. The team accepted Microsoft.Extensions transitive packages as a platform constraint and added a fitness test against the restored package graph.
- The README moved from a stub to a real product entry point with positioning, feature summary, GDPR notice, quick start, architecture link, and project structure.
- The getting-started guide verified examples against real source code. It corrected type shapes, JSON property names, routes, MCP tool names, Keycloak setup, and Windows command paths.
- The sample project became a true external-consumer reference. It demonstrates one-line DI, commands, queries, event subscription, local read-model updates, MCP configuration, and forward-compatible event handling comments.
- The sample subscriber discovered and fixed the real DAPR CloudEvent wrapper shape. Tests now exercise the CloudEvent transport and `cloudevent.id` idempotency pattern.
- CI compatibility stayed visible. TreatWarningsAsErrors issues were fixed rather than ignored, and the sample added targeted tests for event handling and store behavior.

Alice (Product Owner): "The product win is that the developer journey is tangible now. A consumer can see the package, read the guide, and run a sample."

Winston (Architect): "The architecture win is that public integration did not become a back door into CommandApi, Server, Projections, or EventStore internals."

## Challenges

- Source-contract drift appeared repeatedly. Dev notes and planning examples used stale names or shapes such as `search_parties`, `partyType`, and positional `PersonDetails`, while source code used `find_parties`, `type`, and init-only properties.
- NFR28 was too literal for the .NET shared framework reality. The original dependency threshold did not map cleanly to restored Microsoft.Extensions packages, requiring an explicit product/architecture decision.
- Documentation examples required source verification, not memory. Routes, auth setup, JSON payloads, MCP endpoint behavior, and sample commands all needed checks against actual code.
- The sample crossed into Epic 7 concerns earlier than the roadmap suggested. Event subscription, CloudEvent wrapper handling, subscriber scope, idempotency, and forward-compatible event comments became necessary for a useful sample.
- DAPR pub/sub scoping was easy to miss. The sample needed `sample` app-id authorization in the local pub/sub component or the reference subscriber would look valid but not receive events.
- Analyzer hygiene shaped implementation. CA1062, CA2007, NU1510, and xUnit1030 surfaced in story records and required fixes across client and sample tests.
- GDPR communication stayed prominent, but compliance behavior is still deferred. The docs warn clearly, yet consumers could still misread the sample as production-complete for regulated data if they ignore the warning.

Charlie (Senior Dev): "The pattern is obvious: every public-facing example must be source-verified. Otherwise docs and samples drift faster than code."

Paige (Technical Writer): "I agree. The guide is strongest where it names actual files, actual endpoints, and actual commands. Anything copied from planning needs verification."

## Key Lessons

1. Public integration artifacts are executable contracts.
   Evidence: Client tests, sample tests, and documentation examples all had to match actual REST routes, JSON conventions, DI behavior, and MCP tool names.

2. Dependency NFRs need platform-aware wording.
   Evidence: Story 6.1 found that Microsoft.Extensions transitive packages exceeded the literal count, then resolved the issue with a platform-constraint decision and fitness test.

3. Documentation quality depends on source verification.
   Evidence: Story 6.2 corrected MCP tool names, command JSON property names, Keycloak flow, PowerShell examples, and C# type initialization after checking source.

4. Samples expose integration truth earlier than planning documents.
   Evidence: Story 6.3 forced CloudEvent wrapper handling, subscriber app-id scoping, event envelope deserialization, and idempotency tests into working code.

5. Boundary fitness should follow every new public surface.
   Evidence: Story 6.1 added client boundary tests; Story 6.3 kept the sample decoupled from server internals through local event-envelope handling.

6. GDPR warnings must remain attached to every onboarding path.
   Evidence: README, getting-started, and sample comments all carry the MVP warning and defer full compliance behavior to later work.

## Technical Debt And Carry-Forward Items

- Planning artifacts still contain older MCP naming and search-scope language in places. Public docs are better aligned, but PRD/epics/architecture should be reconciled.
- No dedicated no-PII logging regression suite is recorded.
- No named composite-event projection replay suite is recorded.
- Onboarding claims should be validated through repeatable fresh-run evidence, not just asserted in prose.
- Event subscription guidance moved into the sample, but the reusable subscriber documentation belongs with Epic 7 materials and should keep the CloudEvent wrapper details.
- The NFR28 platform-constraint decision should be visible outside the story record so future agents do not reopen the same issue.
- Projection freshness guidance is still scattered. It should be framed consistently for REST, client, MCP, and event consumers.
