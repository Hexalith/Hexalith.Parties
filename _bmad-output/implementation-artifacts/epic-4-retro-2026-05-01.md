## What Went Well

- The composite aggregate pattern was validated. `CreatePartyComposite` and `UpdatePartyComposite` both emit multiple events from a single command while preserving atomicity.
- Validation-first implementation paid off. Story 4.2 separated Phase 1 validation from Phase 2 event emission, which made D12 all-or-nothing behavior understandable and testable.
- Duplicate handling became explicit rather than accidental. Duplicate additions are skipped and surfaced in `CompositeCommandResult.Skipped`; invalid update/remove IDs and conflicts are rejected.
- The test matrix did real risk reduction. Story 4.3 added create, update, no-op, duplicate, conflict, type mismatch, payload boundary, preferred-channel, and nullable-field coverage.
- Review feedback improved the design. Story 4.1 removed PII from operation summaries and added blank sub-operation ID validation. Story 4.2 made duplicate update/remove outcomes visible and deterministic.
- REST validation respected the boundary. FluentValidation enforces structure and payload shape while aggregate rules remain in the aggregate.
- Complete updated state responses were solved without relying on an eventually consistent projection read in the same request.

Alice (Product Owner): "The product win is that AI agents and API consumers can now ask for meaningful complete operations instead of choreographing several fragile calls."

Winston (Architect): "The architecture win is that the aggregate remains the source of truth for domain decisions. That is the line we needed to protect before MCP work."

## Challenges

- The first `CreatePartyComposite` implementation looked simple in story scope but review found important gaps: PII in applied messages, blank sub-operation IDs, and deployment-configurable payload limits.
- `UpdatePartyComposite` had real combinatorial complexity. Conflict detection, state lookups, payload deduplication, nullable patch fields, preferred-channel behavior, and type-specific details all had to work together.
- The result-payload path in Story 4.4 proved unstable as a shared EventStore contract direction. Later Epic 5 records show obsolete `ResultPayload` handling was removed and REST/API assertions were realigned with the current EventStore API.
- Static configuration for `PartyAggregate.MaxSubOperations` created test interference risk. Story 4.3 had to disable test parallelization in the server test assembly to stabilize cases that mutate the shared setting.
- The API needed enriched success payloads, but the existing command dispatch result shape was originally too narrow. That forced a design decision at a shared pipeline boundary, which later needed correction.
- Some Epic 3 carry-forward items were easy to overlook once Epic 4 shifted back to aggregate command work.

Charlie (Senior Dev): "I was too comfortable with the initial `ResultPayload` direction. It solved the immediate response problem, but it leaked into the shared pipeline in a way later work had to unwind."

Amelia (Developer): "That is a system lesson: when an endpoint needs richer response data, shared pipeline changes need extra scrutiny and downstream validation."

## Key Lessons

1. Composite command behavior needs a planned matrix before implementation.
   Evidence: Story 4.3 caught and documented edge cases that would be easy to miss if tests were written only after the handler shape emerged.

2. Aggregate-owned domain validation is the right boundary.
   Evidence: REST validators validate shape, while duplicate handling, entity existence, conflicting operations, and type mismatch stay in `PartyAggregate`.

3. Operation summaries must be treated as a privacy surface.
   Evidence: Story 4.1 review removed derived display-name values and kept applied/skipped/rejected strings ID/type-oriented.

4. Shared command pipeline extensions need compatibility checks against downstream consumers.
   Evidence: Story 4.4 introduced enriched response plumbing, while Story 5.1 later removed obsolete `ResultPayload` assumptions.

5. Static mutable domain settings create test isolation risk.
   Evidence: `MaxSubOperations` tests needed assembly-level parallelization control.

6. Write-side response assembly can avoid projection consistency risk, but it must not become a second projection model.
   Evidence: Story 4.4 mapped updated `PartyDetail` from write-side state and events, then review fixes restored missing `CreatedAt`.

## Technical Debt And Carry-Forward Items

- No dedicated no-PII logging regression guard is recorded for command and query paths.
- Composite-to-projection replay coverage is not clearly closed as an Epic 4 item.
- Story 4.4 documentation contains a now-stale implementation direction around `ResultPayload`; later Epic 5 notes say that path was removed as obsolete.
- `PartyAggregate.MaxSubOperations` remains mutable static state; test isolation is controlled, but the design should stay visible when adding more payload-limit tests.
- Enriched response assembly should be periodically reviewed against `PartyDetail` projection behavior so write-side and read-side shapes do not drift.
- Projection search scope wording was clarified in planning artifacts after Epic 3, but later Epic 5 work appears to extend email/identifier discovery. That should be reflected deliberately in roadmap language.

## Significant Discovery Check

Amelia (Developer): "We have one significant discovery from Epic 4."

Significant discovery:

- The enriched response payload approach changed after Story 4.4. Epic 4 introduced `ResultPayload`-style plumbing, but Epic 5 records show obsolete EventStore `ResultPayload` contract usage was removed and composite REST/API assertions were realigned.

Impact:

- This does not invalidate the composite aggregate design.
- It does mean future retrospectives and planning should treat Story 4.4's original response-plumbing notes as historical context, not current architecture.
- Any new REST, MCP, or client response work must verify the actual current command result path before copying Story 4.4 patterns.

Recommended action:

- Update architecture/story notes or add a short ADR-style clarification for the current enriched response strategy.
