## What Went Well

- The spike made uncertainty visible. Story 12.0 recorded a partial verdict, named the query/projection blockers, and still gave enough evidence to proceed without editing the EventStore submodule.
- The two-wave structure worked. Server topology and boundary changes landed before consumer migrations, so the client, MCP, admin portal, picker, sample, and docs had a real target.
- Fitness tests became architecture memory. Guardrails now check retired REST/MCP surfaces, actor-host boundaries, access-control posture, topology resources, forbidden tokens, and source-text pitfalls.
- Review decisions were resolved rather than hand-waved. Sprint status records multiple code-review passes with explicit accepted decisions, dismissed findings, and deferred items.
- UX follow-through improved during the epic. The admin portal UX spec existed before implementation, and the party picker UX addendum later closed the FR67 design gap.
- The sample and docs moved from historical route examples toward the EventStore gateway, separate `parties-mcp` host, typed client configuration, and subscriber-owned event handling.
- The team preserved submodule discipline. Deferred work correctly separates EventStore-owned follow-ups from Parties-owned implementation.

Alice (Product Owner): "The product win is coherence. A developer or AI agent no longer has to learn one route shape for Parties and another for the platform."

Winston (Architect): "The architecture win is stronger: Parties now validates EventStore as the platform entry point, rather than bypassing it."

## Challenges

- The spike started with proof gaps. Command routing, event persistence, query routing, and projection response needed later stories to turn static analysis into working evidence.
- Planning artifacts drifted behind implementation. The sprint change proposal and story files reflect the pivot more accurately than parts of the PRD and architecture document.
- The UI migration carried inherited debt. Admin and picker rewrites removed retired transport paths but left follow-ups around metadata, cache ownership, cancellation taxonomy, option validation, auth token rotation, and paging bounds.
- Review scope was large. Several stories closed only after many review patches and decisions, which shows both quality discipline and initial story complexity.
- Contract timing was delicate. Stories 12.7, 12.8, and 12.9 repeatedly depended on accepted Story 12.5/12.6 evidence to avoid publishing guessed runnable contracts.
- Deployment validation became broader than topology. Story 12.10 also touched xUnit v3 migration and access-control semantics, creating cross-cutting follow-up audit work.
- Deferred-work volume is now high enough to need triage. Without ownership grouping, the file can turn from useful memory into background noise.

Dana (QA Engineer): "The pattern is clear: every time we tightened a boundary, some old assumption showed up in tests, docs, or UI state. That is exactly what the tests are for."

Paige (Technical Writer): "And the documentation lesson is the same one in another coat: publish accepted contracts, not optimistic ones."

## Key Lessons

1. Architectural pivots need executable boundaries, not just ADR wording.
   Evidence: The actor-host no-REST/no-MCP posture, DAPR access-control rules, typed client boundary, and topology shape are now guarded by tests.

2. A partial spike can still be useful if it is honest.
   Evidence: Story 12.0 did not claim full runtime proof. It classified blockers and made follow-up work explicit, which let later stories close the evidence gap.

3. Consumer migrations should wait for a frozen transport contract.
   Evidence: Admin, picker, sample, and docs work all depended on accepted Story 12.5/12.6 command/query and MCP evidence.

4. Source-text guardrails need parsers or comment/string stripping.
   Evidence: Multiple reviews improved brittle forbidden-token scans by excluding `bin`/`obj`, stripping comments and strings, and narrowing paths.

5. UI correctness is part of platform correctness.
   Evidence: Tenant switch clearing, stale-response suppression, privacy-safe links, token/context forwarding, disabled/read-only behavior, and encoded rendering were necessary to make the new topology safe for users.

6. Deferred work should be treated as backlog input, not a side channel.
   Evidence: Important follow-ups now live in `deferred-work.md`, including EventStore aggregate dispatch, metadata/cache contracts, Gateway routing tests, xUnit v3 audit, deployment migration notes, and topology validation hardening.

## Technical Debt And Carry-Forward Items

- PRD and architecture need a pivot reconciliation pass. Current story records are more accurate than older sections describing Parties REST, in-process MCP, and solution structure.
- `deferred-work.md` needs triage into owned follow-up stories or explicit won't-do decisions.
- EventStore-owned follow-ups remain visible: async derived `DomainResult` dispatch behavior and Gateway invalid-shape routing need separate handling outside Parties scope.
- Admin portal follow-ups include metadata fidelity, not-modified caching behavior, typed-client preference coverage, option validation design, GDPR coordinator state safety, error taxonomy, and safe link hardening.
- Picker follow-ups include `HttpRequestException.StatusCode` classification, token-provider rotation behavior, page upper bound policy, stale selection audit, and replacing fragile privacy substring assertions.
- Documentation follow-ups include command/query envelope conventions, `tenant-a.parties.events` topic naming, cross-shell snippet consistency, and Tenants integration migration notes for `commandApiAppId: eventstore`.
- Test infrastructure follow-ups include a cross-cutting xUnit v3 fixture audit and additional topology/access-control negative fixtures if future topology changes expand the matrix.
- Epic-level status remains `in-progress` in sprint status even though all Epic 12 stories are done; this workflow updates only the retrospective key.
