## What Went Well

- Epic 2 reused Epic 1's foundation effectively. Command/event contracts, PartyState collections, DomainResult patterns, FluentValidation, and ProblemDetails handling were extended rather than reinvented.
- Preferred contact channel behavior was corrected to be per-type, which protects FR11 and avoids clearing unrelated preferred channels.
- Idempotency was applied consistently. Duplicate contact channel and identifier additions return NoOp, matching the D10 retry-safety decision.
- Test coverage matured across the epic. Missing coverage in early stories became explicit follow-up work rather than hidden risk.
- Story 2.3 converted the earlier review lesson into durable tests: identifier handlers, null-state paths, not-found rejections, duplicate handling, and aggregate-size contact channel behavior.
- Story 2.4 ended with endpoint-level tests for all new command routes, including accepted, validation, and domain rejection behavior.
- File list and sprint-status drift improved after review findings made artifact accuracy part of completion quality.

Dana (QA Engineer): "The best pattern was visible remediation. Review findings were not just acknowledged; they were converted into tests and story updates."

Charlie (Senior Dev): "The per-type preferred-channel fix was the right kind of design correction. Small surface area, high semantic impact."

## Challenges

- Story 2.1 had the most important domain defect: `UpdateContactChannel` could violate the preferred-channel invariant when a preferred channel changed type while staying preferred.
- Several story records initially overclaimed validation status or had stale test counts. This repeated the Epic 1 evidence-drift pattern before it was corrected.
- Contact channel behavior had more edge cases than the initial story shape suggested. Preferred status, type changes, duplicate IDs, missing IDs, and null state all needed focused review.
- Story 2.2 looked simple, but review still found integration-test brittleness around command payload parsing and fake actor state.
- Story 2.4 initially lacked endpoint-level coverage for the five new routes. Controller code followed existing patterns, but the epic needed tests proving those routes behaved correctly.
- API method naming guidance conflicted inside Story 2.4 artifacts: earlier notes recommended `Async` suffix, while final review aligned with the existing controller convention and removed it.
- PersonalData logging protection remains largely inspection-backed for this epic. The controller avoids payload logging, but there is no dedicated regression check named in the story evidence.

Alice (Product Owner): "The risk was not lack of effort. It was confidence arriving before the evidence was complete."

Amelia (Developer): "Agreed. The system improvement is to make evidence synchronization part of the definition of done, not a clean-up activity after review."

## Key Lessons

1. Preferred-state invariants need adversarial tests.
   Evidence: The type-change path in Story 2.1 could leave multiple preferred channels for the same target type until review forced a stronger implementation and tests.

2. "Simple" aggregate commands still need rejection and idempotency coverage.
   Evidence: Story 2.2 identifier commands had fewer behaviors than contact channels, but still required PartyNotFound, IdentifierNotFound, and duplicate NoOp validation.

3. Endpoint patterns are not proven by visual similarity.
   Evidence: Story 2.4 needed tests across all five new routes to validate 202, 400, and 422 behavior.

4. Story artifacts are delivery artifacts.
   Evidence: File list drift, stale test totals, and conflicting completion claims repeatedly surfaced as review findings.

5. Inspection-backed privacy claims should become executable checks when the risk grows.
   Evidence: Story 2.4 verified no PersonalData payload logging by inspection only.

## Technical Debt And Carry-Forward Items

- PersonalData logging regression coverage: no dedicated automated check proves command payload values remain excluded from logs.
- Serializer convention mismatch noted in Story 2.4: command dispatch uses default `JsonSerializerOptions`, while MVC uses camelCase and string enum behavior. This is pre-existing but relevant to API consistency.
- Story 2.4 validation tests cover representative invalid payloads but not every enum and multi-field failure case.
- Architecture observation remains open: `UpdatePartyComposite` includes `UpdateIdentifiers[]`, but standalone identifier update is intentionally absent from PRD scope. Epic 4 should resolve or remove that mismatch before composite update implementation.
- Projection-related limitations from Epic 1 now move to Epic 3: read behavior and search must transition from aggregate-state access assumptions to projection-backed query paths.
