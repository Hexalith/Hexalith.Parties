## What Went Well

- The team followed the readiness correction to reuse and enhance `PartiesAdminPortal.razor` instead of creating parallel list/detail/GDPR pages. This avoided duplicated Admin state and kept route/auth behavior concentrated.
- Route-level security stayed layered: `AuthorizeRouteView`, `[Authorize(Policy = "Admin")]`, AdminPortal internal authorization, safe route-id validation, and no-data-call guards were preserved across browse, detail, create, edit, and GDPR routes.
- The Admin list moved from a partial browse surface to a real operator workflow: debounced search, combined type/active filters, stable server paging, stale last-known rows, keyboard row activation, and recoverable empty states.
- The create/edit story added the only genuinely net-new Admin records surface while respecting the typed client/EventStore gateway boundary and route-id authority.
- The picker work completed the D11/UX-DR7 debt in the right package. AdminPortal kept embedding the picker instead of forking picker internals.
- Test coverage grew in the right places: bUnit for component contracts, direct xUnit executable fallback for sandbox/MTP limits, Playwright typecheck and specs for browser behavior, and guardrail tests for transport/privacy boundaries.

## Challenges

- Planning architecture still sketched separate `PartiesListPage.razor`, `PartyDetailPage.razor`, and `PartyGdprPage.razor` files while live code and readiness analysis required reuse of `PartiesAdminPortal.razor`. Story files had to repeatedly restate this correction.
- Shared `PartyStateBadge` and `DataFreshnessIndicator` primitives still live in the UI host. AdminPortal avoided an RCL -> host reference by implementing local equivalent semantics with drift tests, but this is not a clean long-term sharing model.
- Local browser verification remains environment-blocked. Playwright specs and typecheck were added, but Kestrel socket binding failed in this sandbox, so actual browser runs remain CI/environment verification items.
- Package-fixture and full solution checks were partly blocked by NuGet audit/network or pre-existing package warnings outside the Epic 2 surface. Focused builds and direct test executables passed where the environment allowed.
- Story File Lists drifted more than once, especially around E2E fixtures, UI specimen routes, and automation artifacts. Reviews caught and corrected the records, but this was a repeated process issue.

## Key Lessons

1. Readiness corrections need to be promoted into durable planning docs when implementation proves them, not repeated manually in every story.
2. RCL boundaries matter. Avoiding AdminPortal -> UI host references preserved package direction, but shared UI primitives need a neutral ownership decision before more RCLs reuse them.
3. Server-driven search must carry filters into the query/client contract. Client-side filtering would have broken paging/count semantics and privacy expectations.
4. Browser behavior should be covered with Playwright specs even when local execution is blocked, but story records must clearly separate typecheck/spec discovery from actual browser runtime evidence.
5. Route-id authority, bounded event details, and safe validation mapping are reusable patterns for future regulated forms.

## Significant Discoveries Affecting Epic 3

1. The architecture document's file-level AdminPortal tree was stale for Epic 2 implementation reality.
   Impact: Epic 3 should continue reusing the existing AdminPortal route/component structure unless a dedicated refactor story deliberately changes it.

2. AdminPortal command/query adapters now include create/edit command methods and filtered search.
   Impact: API docs and generated inventory needed updates so future stories use the actual adapter surface instead of injecting lower-level clients directly into Razor components.

3. The picker public contract has changed from "basic custom element selector" to a full active-descendant combobox.
   Impact: Future forms should rely on the picker package contract and bounded `party-selected` event rather than adding local keyboard or DOM-event workarounds.

4. Erasure verification is still blocked by the EventStore D7 contract.
   Impact: Epic 3 can implement most DPO operations, but verification report completion requires explicit cross-submodule approval and backend work.
