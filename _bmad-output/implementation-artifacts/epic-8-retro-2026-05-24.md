## What Went Well

- The team reused the existing picker instead of building a second UI package.
- The scoped risk acceptance kept the dependency status honest: every picker story was allowed under explicit conditions without marking the broader EventStore-fronted client/gateway contract satisfied.
- The implementation kept all data access inside `Hexalith.Parties.Picker` and `IPartiesQueryClient`.
- Durable host selection was narrowed to party id through `SelectedPartyId` and `SelectedPartyIdChanged`; display data remains preview-only.
- The picker now resolves host-provided selected ids through the typed query boundary and keeps the durable id stable when display lookup fails.
- Stale-response suppression became broad enough to cover query, selected id, context, auth context, token presence, request customizer identity, disabled/read-only state, and picker options.
- Accessibility and localization were treated as product behavior, not late polish.
- Privacy and boundary guardrails became executable: tests inspect raw markup APIs, browser storage, transport symbols, JavaScript event payloads, package references, and unsafe token handling.

Alice (Product Owner): "The product win is that consuming applications get a selector with a clear identity contract instead of another place where display names become accidental keys."

Winston (Architect): "The architecture win is that risk was accepted without erasing the dependency. That distinction matters for future contract freeze work."

## Challenges

- The epic started blocked by the EventStore-fronted Parties client/gateway scheduling gate. Work required a correct-course decision before implementation could proceed.
- The `.NET` callback model needed source compatibility with preview display data while still making party id the only durable host value.
- Stale selected-display state had more branches than the initial shell implied, especially when only `SelectedPartyId` changed.
- Host token changes created a privacy tradeoff: token-value fingerprinting helped invalidate stale contexts but was removed because token values must not participate in component signatures.
- Blazor accessibility semantics needed review fixes to complete the combobox/listbox pattern.
- JavaScript event dispatch needed a defensive whitelist so only `partyId`, `partyType`, and `status` leave the component.
- Broader solution regression attempts continued to expose unrelated failures outside the picker scope, so the trustworthy closure signal remained the focused picker and affected client suites.

Murat (Test Architect): "The recurring quality issue was not missing happy-path behavior. It was edge-state leakage: stale data, token-derived state, callback payload width, and UI semantics."

Charlie (Senior Dev): "The useful constraint was the guardrail suite. It kept every story from quietly reaching into a more convenient but forbidden boundary."

## Key Lessons

1. Scoped risk acceptance works only when the residual risk remains visible.
   Evidence: The dependency record still says the full contract is not globally satisfied, and each story repeats the binding conditions.

2. Durable identity and preview display data need separate API paths.
   Evidence: `SelectedPartyIdChanged` carries the durable id, while `SelectedPartyChanged` remains preview-oriented for source compatibility.

3. Stale-response handling must include selected-display lookup, not just search.
   Evidence: Review added coverage for stale selected-display responses after selected-id-only changes.

4. Privacy-sensitive context invalidation should use host version keys, not token-derived fingerprints.
   Evidence: Story 8.6 removed token-value fingerprinting and relies on `AuthContextKey` plus token presence/delegate identity.

5. Accessibility patterns need the same review rigor as transport boundaries.
   Evidence: Story 8.5 review added `role="combobox"` and `aria-haspopup="listbox"` to complete the input/list relationship.

6. Documentation examples must use the durable contract first.
   Evidence: Post-retro documentation audit updated the Blazor usage example to bind `SelectedPartyId` and `SelectedPartyIdChanged`.

## Significant Discovery Check

Significant discovery 1:

- The existing picker plus `IPartiesQueryClient` was sufficient for Epic 8 under scoped risk acceptance.

Impact:

- The formal contract can be shaped by observed picker behavior instead of abstract planning alone, but it still needs explicit acceptance before the dependency is globally satisfied.

Significant discovery 2:

- Token-value fingerprinting is not an acceptable privacy mechanism for picker context signatures.

Impact:

- Hosts must use `AuthContextKey` to represent non-sensitive auth-version changes when token provider/customizer instances are stable.

Significant discovery 3:

- The DOM event boundary needs an independent JavaScript whitelist, even when the .NET DTO is narrow.

Impact:

- Future custom-element integrations should treat JavaScript dispatch as a separate privacy boundary with tests.

Epic update required: No fundamental Epic 9 direction change is required. Future contract work should update the dependency record and reconcile the picker bridge when the formal client/gateway contract is accepted.
