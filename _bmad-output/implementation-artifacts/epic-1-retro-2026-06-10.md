## What Went Well

- The team consistently used architecture and project-context constraints as guardrails. Examples include Central Package Management, `TreatWarningsAsErrors`, no public API on the actor host, root-repository submodule under `references/` discipline, no browser tokens, and no Dapr sidecar on `parties-ui`.
- Security posture improved incrementally and defensibly: host-owned OIDC, server-side cookie session, role-policy routing, fail-closed `party_id` resolution, self-scoped consumer accessor, and host-side consumer policy building blocks.
- Testing evolved with the epic. The final stories had strong bUnit coverage, direct xUnit executable fallbacks for sandbox limitations, deploy validation tests, Playwright discovery/typecheck, and static secret/manifest validators.
- Reviews caught meaningful defects before completion: missing file-list entries, stale test counts, route authorization enforcement, skip-link target correctness, CI build ordering, destructive action empty-confirmation behavior, and secret-validation leakage risk.
- Deployment documentation and manifests were brought into alignment with the new `parties-ui` workload, including the 12-pod topology, non-Dapr UI workload, public UI ingress, OIDC secret reference, and production KMS gate.

## Challenges

- Several story records required post-implementation cleanup of File Lists and Debug Log test counts. This did not indicate behavioral defects in most cases, but it made review harder and increased reliance on git/story cross-checking.
- Local sandbox limits affected verification. `dotnet test` named-pipe creation, Kestrel socket binding, NuGet audit access, and full interactive Playwright execution were blocked in some runs, so agents had to use direct test executables, typecheck/discovery, static validation, and CI deferral.
- Scope boundaries had to be defended repeatedly. Examples: no live OIDC browser verification in sandbox, no profile write before later Consumer stories, no per-screen status remaps, no gateway request-path own-data enforcement before the gateway can carry an end-user principal, and no EventStore D7 contract without cross-submodule approval.
- Some implementation decisions exposed follow-up architecture work rather than code defects. The host-owned `StatusKind` map is not directly consumable by independent RCLs, and the degraded-header handler writes in an `IHttpClientFactory` handler scope rather than the Blazor circuit scope.

## Key Lessons

1. Review workflows should always reconcile git changes, story File Lists, and test-count claims before approval.
2. The UI host can safely provide cross-area foundations, but RCL consumption boundaries need explicit design before Epic 2 and later Consumer work.
3. Projection freshness metadata is the reliable degraded-state signal today; header relay into circuit state is only a building block until a per-circuit bridge exists.
4. Accessibility gates need both component-level tests and browser-level CI. Local sandbox failures should be recorded precisely without weakening the gate.
5. Production deployment security needs source-level validators, not just operator instructions; the OIDC secret and raw credential scanner fixes proved that.

## Significant Discoveries Affecting Epic 2

1. `StatusKind` and shared domain components live in `Hexalith.Parties.UI`, while AdminPortal is an independent RCL.
   Impact: Epic 2 must decide whether to promote status/freshness primitives into a shared UI package or keep mapping at the host composition boundary. Direct RCL reference to the host would violate the intended package direction.

2. `DegradedResponseHeaderHandler` is not yet a live per-circuit degraded-state bridge.
   Impact: Epic 2 data screens should rely on `ProjectionFreshnessMetadata` first and should not assume the header accessor reflects gateway responses in the active Blazor circuit until a bridge is implemented.

3. Host-local `/admin` and `/me` stubs are temporary.
   Impact: Story 2.1 must remove or avoid duplicate `/admin` host routes when AdminPortal routes are added to the router.

4. Live OIDC role claim verification remains environment-dependent.
   Impact: The first environment capable of interactive login should verify the flat `roles` mapper and `RoleClaimType = "roles"` assumption.

5. The interactive Playwright a11y gate is CI-bound where local socket/framework-script limits prevent complete execution.
   Impact: Epic 2 should preserve the CI gate and add AdminPortal-specific bUnit coverage so local work still has useful feedback.
