## What Went Well

- The Epic 4 ConsumerPortal port/adapter rule held across all four stories. ConsumerPortal owns narrow caller-id-free ports; the UI host adapts them to `ISelfScopedPartiesClient`.
- Boundary tests scaled with each feature. Packaging and surface tests continued to forbid UI-host references, direct gateway clients, list/search, `PagedResult<>`, and party-id shaped ConsumerPortal APIs.
- Resource-backed regulated copy became the default implementation habit for consent, export, erasure, and processing transparency.
- Focused validation stayed reliable despite the local `dotnet test`/MTP wrapper issue. Story records consistently captured direct Release builds and compiled xUnit v3 executable runs.
- Senior reviews found high-value issues before completion: active consent withdrawal without a current channel, incomplete export fixture payloads, erasure authoritative-status reconciliation, terminal erasure status persistence, and unsafe processing-summary pass-through.

## Challenges

- Story 5.3 revealed a real backend contract gap. The planned request/status path could not support Consumer cancellation honestly; the implementation needed additive `CancelPartyErasure` and `PartyErasureCancelled` contracts plus authoritative `GetErasureStatus`.
- The same story showed that erasure status cannot be inferred only from `PartyDetail.IsErased`. Cancellable, in-progress, verified, and erased states need the erasure record store or equivalent lifecycle status source.
- The test wrapper remained noisy and misleading: it returned exit code 0 while printing repeated build-failure messages, and direct `dotnet test` still hit local handoff/socket limitations.
- Browser runtime proof for Playwright remained blocked by sandbox socket permissions. Typecheck and spec discovery passed, but runtime route proof could not bind Kestrel locally.
- File-list hygiene improved but did not become perfect. Story 5.3 review still had to add a touched validation test and story-automation artifact to the story File List.

## Key Lessons

1. ConsumerPortal ports are the right long-term pattern. They kept the RCL independent while allowing self-scoped BFF enforcement and focused UI tests.
2. Privacy UX often exposes missing backend semantics. Cancellation needed a real command and status source; pretending request/status was enough would have produced misleading copy.
3. "Bounded metadata only" needs adapter enforcement, not just UI restraint. Story 5.4 fixed unsafe processing-summary pass-through by dropping unsafe backend summaries before they reached the RCL.
4. Fixture fidelity matters. The export fixture needed a complete portability package to prove the browser path would map to ready instead of transient failure.
5. Authoritative reconciliation must win over optimistic assumptions. Story 5.3 fixed cancellation mapping so the UI preserves refreshed pending status unless the backend proves the request is active/cancelled.

## Significant Discoveries

Epic 5 changed the durable architecture in one important way:

- Consumer erasure cancellation required a real additive backend contract (`CancelPartyErasure` command and `PartyErasureCancelled` event) rather than reusing `LiftRestriction` or relying only on request/status reads.
- `GetErasureStatus` needed to return authoritative pending and lifecycle status, not just derive terminal erased state from `PartyDetail`.

These are documentation-impacting implementation learnings and were added to the post-retrospective documentation audit.

## Commitments

- Keep ConsumerPortal independent from the UI host and direct gateway clients.
- Keep Consumer self-service methods no-caller-id at the RCL boundary.
- Do not describe erasure cancellation or status as implemented unless the docs mention the additive command/event and authoritative status query.
- Do not claim Playwright runtime proof when the environment only allowed typecheck/spec discovery.
- Do not start a new Consumer privacy surface without boundary, copy, and PII-filtering tests.
