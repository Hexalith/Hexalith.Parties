## Story

As an operator (or developer) deploying Hexalith.Parties from `main` to a local Kubernetes cluster,
I want the aspirate-generated manifests under `deploy/k8s/` to be sufficient — without manual `kubectl set env`, hand-created Secrets, or out-of-repo `Deployment` resources — to bring the documented topology to `Ready`,
so that Story 9.1's "one-command flow" promise holds reproducibly from a clean checkout, and the full sibling-submodule topology (EventStore, Tenants, **Memories**, **FrontComposer**) plus the Keycloak / Redis infrastructure dependencies promised by FR31a are emitted into the repo rather than improvised on the cluster.

## Architectural Decisions Recorded by This Story

Four locked-in design choices, formatted as compact ADRs (mirror the `architecture.md` D-* style). These are project-level architectural state once Story 9.3 closes; do not re-litigate without an explicit ADR amendment.

### ADR 9.3-1 — Wildcard EventStore Registration Key Shape

- **Status:** Accepted
- **Context:** K8s ConfigMap data keys + Pod env names reject `*` and `|`; the existing `*|party|v1` dictionary key in `src/Hexalith.Parties.AppHost/Program.cs:13-19` is K8s-invalid (2026-05-18 live deploy needed `kubectl set env` workaround).
- **Decision:** Replace the dictionary key with `wildcard_party_v1` (alphanumeric + underscore only). EventStore `DomainServiceResolver.cs:49-66` is verified or extended to match the new shape via a parallel PR in `Hexalith.EventStore`.
- **Rejected alternatives:** (a) List-shape `Registrations__0__*` — requires broader EventStore binder rewrite, larger blast radius. (b) AppHost-side serialization adapter — inverts dependency direction; future EventStore consumers would re-discover the K8s-naming constraint.
- **Consequences:** AC1 may temporarily close `partial` if the EventStore PR slips past 3 working days. The K8s-name-validity constraint is absorbed once by the EventStore platform.

### ADR 9.3-2 — Memories.Server Composed In-Cluster Only

- **Status:** Accepted
- **Context:** The 2026-05-19 sprint change proposal § 4.2 considered carving "configurable external HTTP Memories endpoint" into FR31a. Convergent review identified this as a stealth erosion of FR31a's "single source of truth" principle.
- **Decision:** Memories.Server is composed via `AddProject<Projects.Hexalith_Memories_Server>("memories")`. The in-cluster Service URL (`http://memories:8080/`) is the only endpoint. `EnableMemoriesSearch` controls feature-on, never endpoint location.
- **Rejected alternatives:** Configurable `MemoriesEndpoint` opt-out for managed-external Memories operators — preserved an unscoped second topology.
- **Consequences:** Future managed-external Memories deployments require a new FR amendment + story. FR31a remains enumerative.

### ADR 9.3-3 — Keycloak via Hand-Authored `deploy/k8s/keycloak/` Carve-Out (Path B)

- **Status:** Accepted (with documented contradiction — see Dev Notes § "Known Contradiction")
- **Context:** Aspirate captures Keycloak's randomized admin password into the emitted ConfigMap each regen, breaking AC1 byte-determinism + violating the no-secret-values contract (Story 9.1 Completion Notes #5). Aspirate-native Keycloak emission has been deferred for this reason since Story 9.1.
- **Decision:** Ship `deploy/k8s/keycloak/` as hand-authored Deployment + Service + ConfigMap realm import + admin-password `secretKeyRef`. `regen.ps1` preserves the folder + asserts presence; byte-determinism contract excluded on this subfolder only. JWT `secretKeyRef` patched into consumer Deployments by new `regen.ps1` post-aspirate step (mirrors the existing Dapr-annotation patch at `regen.ps1:159-179`).
- **Rejected alternatives:** (a) Aspirate-native emission (path a) — Keycloak's chart drifts faster than aspirate can track; randomized admin password is a structural mismatch. (b) Aspire-side `secretKeyRef` extension — adds maintenance surface for one tool gap.
- **Consequences:** Two manifest-authoring conventions coexist in the repo (generated + hand-authored). Documented as architectural debt for Epic 10 tool-choice review (see § Known Contradiction).

### ADR 9.3-4 — FrontComposer Composition: Outcome A or Outcome B Carve-Out

- **Status:** Accepted
- **Context:** `FrontComposer.Mcp` is `Microsoft.NET.Sdk` library (no `Program.cs` visible); `FrontComposer.Shell` is `Microsoft.NET.Sdk.Razor` UI components. Neither is a deployable host today. Authoring a new host project from within Story 9.3 is scope creep.

- **Rejected alternatives:** "Stop and escalate" without artifact — left AC3 with no Done state and risked mid-implementation status whiplash.
- **Consequences:** Epic 9 retro gates on Story 9.4 done if Outcome B. Sprint planning visibility preserved.

## Out of Scope

The following are **explicitly NOT this story's job**. Dev must surface a scope-creep flag if any of these creep into a PR draft:

- **Production-grade Redis** (StatefulSet + PVC + AUTH + TLS) — post-MVP cloud story.
- **Production-grade Keycloak** (HA, PostgreSQL backend, external realm export, pinned admin credentials beyond the dev-mode bootstrap) — post-MVP cloud story.
- **Managed cloud Kubernetes** (AKS / EKS / GKE) — explicitly rejected by ADR D-K8s for the MVP scope.
- **Authoring a brand-new FrontComposer host project** — if neither `FrontComposer.Mcp` nor `FrontComposer.Shell` is host-ready today, follow Task 4 Outcome B: carve FrontComposer out by creating Story 9.4 (`9-4-frontcomposer-deployable-host`) and pass AC3 with the carve-out artifact set. Authoring a new host project from within Story 9.3 is explicit scope creep.
- **Managed-external Memories HTTP endpoint** — AC2 commits to in-cluster `memories` Service as the **single** endpoint. There is no "configurable HTTP endpoint" escape hatch in FR31a or in this story. If a future operator persona needs managed-external Memories, that is a new FR amendment + its own story; do not preserve the `MemoriesEndpoint` configuration knob "just in case".
- **`Memories.EventStore` / `Memories.Redis` as separate state-plane projects** — Memories owns its own state plane; this story only composes `Memories.Server` (the service host) into the Parties topology.
- **AdminPortal / Picker UI deployment** — those are a separate Epic concern (UI delivery), even though FrontComposer.Shell is referenced from AdminPortal.
- **Modifying the Story 9.1 / 9.2 categories or test files** — Story 9.3 is **additive** to those. Existing tests stay green unchanged; new tests live in new `[Fact]`/`[Theory]` blocks or a sibling test class.
- **Wiring the validator into `deploy-local.ps1` as a pre-flight** — Story 9.2 explicitly deferred this; 9.3 inherits that deferral (the new `kubectl apply --dry-run=server` step in `deploy-local.ps1` for `resiliency.yaml` is a single targeted post-install probe, not a generalized pre-flight invocation of `validate-deployment.ps1`).
- **Replacing aspirate** with Aspire's native `AddKubernetesEnvironment` publisher — ADR D-K8s pins aspirate; the orthogonal `PUBLISH_TARGET=k8s` block in `Program.cs` (lines 184-205 of the current source) stays untouched.
- **Inviting Memories / FrontComposer submodule code edits beyond the AppHost composition** — if either submodule needs a host-ready surface or a publishable flag fix, surface as a parallel commit/PR in the submodule (per Project Context rule: "treat sibling directories … as root-level submodules; avoid editing them unless the task explicitly crosses that boundary").

### Known Contradiction — Generator-of-Truth vs Hand-Authored Carve-Out (TRIZ Contradiction 7)

**Diagnosis surfaced during party-mode review (Dr. Quinn, 2026-05-19):** Story 9.3 commits to aspirate as the single generator-of-truth for `deploy/k8s/` (ADR D-K8s) AND ships a hand-authored `deploy/k8s/keycloak/` carve-out (ADR 9.3-3). These two are in TRIZ-Contradiction-7 tension — the same directory must be both "fully generated, byte-deterministic, do not hand-edit" (Story 9.1 contract) and "this one subfolder is hand-authored, do hand-edit, carved out of byte-determinism" (Story 9.3 AC4). The story does not resolve the contradiction; it bounds and documents it.

**Why it's accepted for Story 9.3:** aspirate at `9.1.0` cannot express (a) Keycloak admin credential via stable `secretKeyRef` (it captures a randomized value into the ConfigMap each regen — Story 9.1 Completion Notes #5), (b) `WithRealmImport` realm-bootstrap ConfigMap, (c) consumer-side JWT `secretKeyRef` envFrom from an Aspire `WithEnvironment(...)` literal. These three are the precise tool gaps. The hand-authored carve-out is the boring-tech workaround until aspirate gains expressive completeness OR is replaced.

**Reinforcing-loop risk (systems-thinking lens):** every aspirate gap → hand-authored carve-out → carve-outs drift from generator over time → next deploy needs another manual fix → another carve-out story (9.5, 9.6, 9.N). Story 9.3 closes ONE iteration of this loop; it does not break the loop. The detection mechanism (AC8 trait-gated live-cluster test + `K8sTopology-MissingService` lint) is a tripwire, not a fix.

**Architectural debt parked for Epic 10 tool-choice review:** at Epic 10 (or whenever the next aspirate-gap surfaces), revisit one of three resolutions:
1. **Extend aspirate** — contribute upstream patches that emit Keycloak with `secretKeyRef`, preserve realm import, and translate `WithEnvironment(SecretKeyRef)`. Single source of truth restored.
2. **Replace aspirate** — evaluate alternatives (Pulumi, kustomize templates from a Roslyn source-generator over the AppHost graph, Helm charts emitted by a custom Aspire publisher). Two-tool boundary made explicit.
3. **Formally split the topology** — `deploy/k8s/` is aspirate-generated; `deploy/infra/` (new) is hand-authored for infrastructure dependencies (Keycloak, Redis, future Postgres/Kafka). Each tool has one job; carve-outs become a directory boundary, not a per-subfolder exception.

Track this as a separate ADR amendment ticket, not as Story 9.3 dev work. The dev agent inherits the diagnosis, not the resolution.
