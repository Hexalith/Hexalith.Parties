## What Went Well

- Epic 10 converted backend capabilities into operator-facing and adopter-facing workflows without duplicating Tenants administration.
- FrontComposer/Blazor became the consistent frontend direction, replacing earlier TypeScript/standalone-portal assumptions.
- The team used privacy-safe UI rules repeatedly: encoded rendering, no raw markup paths, no PII in storage keys/logging/event payloads, bounded ProblemDetails handling, and safe filenames.
- Review pressure caught real defects. Examples include stale tenant/detail responses, uncaught transport failures, 204/304/null payload handling, focus and keyboard regressions, fail-open rich-search probing, EventStore command/query envelope mismatches, and picker host-auth propagation gaps.
- The `10-1-1` follow-up story made explicit deferrals visible instead of pretending Story 10.1 fully solved FrontComposer, Tenants, and rich-search integration on the first pass.
- The picker ended with a cleaner consumer contract: selected party id is durable, preview fields are non-sensitive, host auth stays host-owned, and JavaScript embedding does not create a second frontend stack.
- Epic 12 successor stories were used pragmatically to close Epic 10 historical gaps once the EventStore-fronted architecture pivot landed.

Alice (Product Owner): "The product win is that administrators and consuming app developers now have real surfaces, not just API stories."

Winston (Architect): "The architecture win is boundary discipline. Frontend work adapted to Tenants and EventStore rather than building around them."

## Challenges

- Epic sequencing became non-linear. Sprint status records Epic 11 and Epic 12 around Epic 10, and Epic 10 closure depends on successor evidence from Story 12.7 and Story 12.8.
- The initial admin portal scope had several intentional partials: Tenants-derived auth, rich-search capability detection, FrontComposer Fluent UI adoption, query transport, and coordinator wiring.
- Metadata fidelity regressed or stayed unresolved during the typed query transition. Degraded-search headers, ETag/not-modified payload ownership, and typed-client metadata shape remain deferred.
- GDPR UI state management was more subtle than simple button wiring. Erasure status, consent state, export behavior, party/tenant switches, stale continuations, and in-flight cancellation all needed defensive handling.
- EventStore gateway contracts were still moving while Epic 10 closed. Some GDPR operation tests were intentionally skip-marked or rewritten ahead of later EventStore/Parties client convergence.
- Full validation evidence is uneven across story records. Some stories have full solution proof; Story 10.3 later notes focused suites passed while a full build blocker was unrelated.
- Deferred work has accumulated across AdminPortal, picker, tenant authorization, search, and projection determinism. The list is useful, but it now needs prioritization.

## Key Lessons

1. Frontend stories need the same boundary rigor as backend stories.
   Evidence: The strongest Epic 10 fixes were about Tenants authority, EventStore query ownership, and Parties client boundaries, not visual polish.

2. Privacy-safe UI is a systems property.
   Evidence: Story 10.2 and Story 10.3 had to coordinate rendering, storage, filenames, custom event payloads, auth cleanup, stale responses, and bounded errors.

3. Successor evidence can close historical scope, but only if the dependency is explicit.
   Evidence: Story 10.1.1 closed remaining AC3 work against Story 12.7 evidence and called out successor files for traceability.

4. Metadata contracts deserve first-class design.
   Evidence: Degraded-search status, stale data age, ETag/not-modified behavior, and typed-client metadata remain deferred because the query contract was not stable enough during the rewrite.

5. Component embedding contracts must include auth and context-change semantics from the start.
   Evidence: The picker review found that host auth was presence-checked but not applied to outgoing queries until the typed client request hook was added.

6. Retrospective timing matters.
   Evidence: Running Epic 10 retro after Epic 12 reveals useful hindsight, but it also blurs "next epic preparation" because the nominal next epics are already complete.

## Significant Discovery Check

Amelia (Developer): "Epic 10 did not invalidate the broader product direction. It did reveal where the next layer of design pressure lives."

Significant discovery 1:

- Administration UX depends on EventStore/FrontComposer query metadata, not only payload shape.

Impact:

- Without a stable metadata contract, the portal cannot reliably distinguish fresh, stale, degraded, not-modified, and local-only states across all query paths.

Significant discovery 2:

- Tenant and auth transitions are the highest-risk frontend state boundary.

Impact:

- UI components need context signatures, cancellation, and stale-response suppression as standard patterns.

Significant discovery 3:

- GDPR operations behave like long-lived compliance state machines.

Impact:

- UI actions must re-read authoritative state, avoid inferred completion, and preserve audit clarity without exposing PII.

Significant discovery 4:

- Embedding is not just packaging.

Impact:

- The picker must carry auth, privacy, selection, cleanup, accessibility, and host interoperability semantics as part of its public contract.

Epic update required: No fundamental product pivot is required. The important update is to treat metadata, stale-context handling, and deferred-work prioritization as release-readiness work, not polish.
