## What Went Well

### Breakthroughs worth replicating

1. **Story 4.7 turned the MCP boundary into executable architectural fitness, not policy text.** The story added project-level fitness asserting the MCP production references are exactly `Hexalith.Parties.Client`, `Hexalith.Parties.Contracts`, `Hexalith.ServiceDefaults`, and `ModelContextProtocol` — and **only** these. Source-level fitness rejects raw `ProblemDetails`, exception `.Detail`/`.Message` use, claim-payload handling, and authorization/token wording in tool responses. The boundary is now a test the team cannot accidentally cross, rather than a checklist that decays. This is the strongest concrete realization of D11 (MCP must translate, not own domain logic) since the architecture phase.

2. **Story 4.1 removed `get_party_name_at` from the advertised MCP surface as the very first epic act.** The temporal-compatibility tool stayed internal/future-only; the agent-visible set is now exactly `find_parties`, `get_party`, `create_party`, `update_party`, `delete_party`. Doing this in 4.1 — before 4.2–4.6 implemented behavior — meant every downstream story could assume the canonical five-tool roster. Pattern: lock the public taxonomy before implementing the behavior.

3. **Soft-deactivation semantics shipped as explicit response keys, not implicit behavior (Story 4.6).** `delete_party` responses include `operation = soft-deactivation` and `gdprErasurePerformed = false` as first-class fields, plus an idempotent already-inactive path that returns success without re-dispatching. This makes the MVP-vs-GDPR-erasure distinction discoverable by AI agents without prose parsing — exactly the kind of structured cue agents can reason on. Pre-empts an entire class of agent-behavior bugs.

4. **Complete-response contract proven test-side, not assumed (Story 4.8).** `create_party`, `update_party`, and `delete_party` happy paths are covered with full gateway-payload responses verified end-to-end, and `get_party` / `find_parties` include projection freshness metadata where applicable. This closes a gap from the Epic 1.9 era: agents do not need a follow-up `get_party` to know the resulting state. Latency target also asserted (<1s MVP budget for in-process calls).

5. **Tenant/context fail-closed proven per-tool, not per-suite (Story 4.7).** Cross-tool dispatch coverage demonstrates each of `find_party`, `create_party`, `update_party`, and `delete_party` fails closed **before** any query/command-client access when tenant or user context is missing. This is the Epic-3 lesson "guards must be proven per-method, not per-client" (recurring review feedback in Epic 3) applied proactively at the MCP boundary.

6. **Validation-first composite create/update (Stories 4.4 + 4.5).** `create_party` and `update_party` reject oversized payloads **before** any command dispatch, and validate caller-supplied party IDs before the EventStore gateway is called. Patch semantics in 4.5 merge against current details so omitted fields are absent rather than nulled — the patch-vs-replace ambiguity that bit aggressive AI clients in the older Epic-4 (composite) era is now structurally impossible at the MCP boundary.

7. **Error mapping verified for content, not only for status (Stories 4.7, 4.8).** Error mappings are inspected for safe wording — omitting personal data, raw payloads, secrets, and infrastructure details — not just for "did we return a problem detail." This raises the floor for agent-safe diagnostics from "structurally a problem detail" to "no exfiltration risk in the prose."

8. **Test-count growth as a coverage signal, not a story budget.** The cumulative MCP test count climbed 21 → 23 → 20 → 26 → 32 → 42 → 48 → 52 across the epic (slight 4.3 dip reflects filter-narrowing on `get_party`, not test loss). Each story added tests rather than rewriting older ones — additive coverage on top of a stable contract. This is the inverse of the "rewrite the test pyramid every story" anti-pattern.

## What Didn't Go Well (systemic patterns, no blame)

### 1. TD-3-A solution-build workaround inherited into every single story

**Stories affected:** 4.1, 4.2, 4.3, 4.4, 4.5, 4.6, 4.7, 4.8 (all 8)

Every Epic 4 story validation command used the override `-p:TreatWarningsAsErrors=false -p:WarningsAsErrors=` because `Hexalith.EventStore` submodule analyzer warnings still get promoted to errors under the solution-level build. Each story includes the same "warning override note" in its Notes section. This is exactly the carry-forward from Epic 3 (TD-3-A / B8) — eight more stories now route around the broken solution build instead of fixing it. Story 9.8 (`solution-build-green-on-clean-clone`) was carved out at the Epic 3 retro on 2026-05-22 specifically to address this, but it is still `ready-for-dev`. The workaround has now propagated through 18+ stories across Epics 3, 4, 6, 7. CI cannot be a quality signal until 9.8 lands.

### 2. Story records are summary-only — no struggle log, no Dev Agent Record narrative

**Stories affected:** 4.1, 4.2, 4.3, 4.4, 4.5, 4.6, 4.7, 4.8 (all 8)

Every Epic 4 story file is in the compact orchestrator form: Status, Scope/Summary, Validation, Notes (warning override). There is no "Senior Developer Review (AI)" section, no party-mode/advanced-elicitation trace, no struggle log, no Deferred Decisions list. By contrast, Epic 3's 3.1 / 3.2 stories explicitly recorded two-pass review + Senior Dev Review (AI) timestamps, and Epic 1 stories named what they were *not* doing. **The MCP epic's evidence trail is the test count and the commit log — not the story records themselves.** This is fine when nothing surprises you, but it means the retrospective has no documented near-misses to learn from, and any HIGH/MEDIUM findings caught during automated review are not preserved in the story file for future readers. If 4.7's boundary fitness regresses later, there is no narrative explaining *why* each constraint was added.

### 3. Story 4.4 omits the FluentValidation/oversized-payload validation evidence from its public summary

**Story affected:** 4.4

4.4's Scope mentions "Oversized create payloads are rejected with a structured MCP validation error before any command is dispatched," but the Notes/Validation section does not enumerate which test (or which dispatch test file) covers the payload-limit guard. The Epic 4 cumulative count proves the test exists (26 dispatch tests by 4.4), but the story file by itself does not pinpoint the assertion. Future regression would have to be discovered by failing the whole dispatch test class, not by name. Same shape applies to 4.5 (`update_party` oversized payload rejection).

### 4. `find_parties` MVP-only-display-name guard relies on prose, not on structural rejection

**Story affected:** 4.2

4.2 documents that email/identifier/semantic/graph/temporal search are *not evaluated* in MVP, and the dispatch tests cover validation errors for malformed date filters. But the "do not claim unsupported search fields were evaluated" acceptance criterion (epic line 1559) is asserted via observable response shape, not via a structural test that fails the suite if a future contributor wires up email or identifier search to `find_parties`. Long-term, this is a fitness-test-shaped gap: when Epic 5 (already done — events) or a future search-feature story extends `Hexalith.Parties.Projections`, there is no guardrail preventing it from being reflected at the MCP boundary without intentional taxonomy review.

### 5. Composite payload limit is still static mutable state — Epic-4-old lesson not re-imported

**Stories affected:** 4.4, 4.5 (indirectly)

The 2026-05-01 retro (older renumbered Epic 4) flagged `PartyAggregate.MaxSubOperations` as static mutable configuration with test-parallelization implications. The current Epic 4 stories now enforce the *MCP-side* oversized-payload guard before dispatch, which is good — but the underlying aggregate-side configuration is still global. If a future test mutates it in the MCP test assembly, Epic 4's oversized-payload tests could become order-dependent. No story in the current Epic 4 re-imported that test-isolation discipline because the older retro was about a different epic by number; the lesson stayed in the old file.

## Significant Discoveries (transferable design principles)

**Epic update required: NO** (Epic 5 already done; Epic 7 / 8 contracts unaffected). The discoveries below are transferable principles, not amendments.

1. **Epic renumbering silently invalidates prior retros for that slot number.** `epic-4-retro-2026-05-01.md` is for "Composite Commands & Advanced Aggregate Logic" — a 4-story epic that no longer exists at slot 4 in the current plan. Anyone reading `epic-4-retro-*.md` chronologically would conflate two different epics. Future renumbering events should either (a) move the old retro into an `archive/` folder, (b) prepend a Stale Predecessor banner to the old file, or (c) include the epic title in the filename (e.g., `epic-4-mcp-retro-*.md`). Captured as **B9**. The same risk exists for any pre-renumbering story file (e.g., `4-1-create-party-composite-aggregate-handler.md` and `4-1-register-bounded-mcp-tool-surface.md` coexist; sprint-status disambiguates by story key, but raw filesystem listing does not).

2. **Story-record minimalism removes the audit trail that retrospectives are built on.** Epic 1 and Epic 3 retros leaned heavily on story-level traces (two-pass review timestamps, Senior Dev Review (AI) HIGH/MEDIUM findings, Deferred Decisions). Epic 4 stories are summary-only. The retrospective therefore had to be reconstructed from cumulative test counts, commit messages, and acceptance-criteria cross-references, rather than from the stories' own narrative. If the team values retro signal quality, the story-record template must require Dev Agent Record + Deferred Decisions sections even when a story is "just" tightening an existing surface. Captured as **B10**.

## Key Takeaways (carry forward)

1. **Lock the public taxonomy in story 1 of the epic.** Story 4.1 removed `get_party_name_at` and pinned the five-tool roster before any behavior story landed. Epic 1 did the same with the rejection taxonomy. Epic 2 did the same with the freshness taxonomy. Epic 3 did the same with the rejection-catalog-to-status mapping. **When this happens, downstream stories never need an "architecture reconciliation" tax.**
2. **Promote policy to executable fitness on the first cross-cutting story of an epic.** Story 4.7 turned "MCP must not reference DAPR actors / projection internals / raw ProblemDetails / claim payloads" from prose into project-level and source-level fitness tests. This is the only mechanism that survives future contributors who did not attend the retrospective.
3. **Per-method / per-tool fail-closed proof is non-optional for boundary surfaces.** 4.7 covers every tool, not one representative tool. This is the inverse of the Epic-3 review-feedback theme "pre-canceled token zero-send proof per method, not per client." Generalize: any safety claim that lives at the *surface* of a boundary must be proven per *element* of that surface.
4. **Structurally signal MVP-vs-future-feature distinctions in the response shape, not the prose.** `delete_party` returns `operation = soft-deactivation` and `gdprErasurePerformed = false` as keys. AI agents read keys, not paragraphs. Apply the same principle to any future MCP tool whose MVP behavior diverges from an obvious user expectation.
5. **Velocity in a "tighten existing surface" epic is not a planning datum.** Epic 4 finished in a single-day commit window because the host scaffold existed. Do not project Epic 4's pace onto Epic 7 / Epic 8 (which are building new surfaces against blocked contract gates). Captured implicitly in the velocity table.
6. **Renumbering invalidates prior retros at the same slot.** `epic-4-retro-2026-05-01.md` belongs to a different epic. Until B9 lands, retro readers must check epic *title*, not epic *number*.
